## 0. 設計思想(動かない前提)

- **ハーネス + 骨格事前固定のハイブリッド** ── 当初の「気質+環境から完全創発」案は採用しない。性格の骨格(気質・性格特性・価値観の骨格・理想自己の方向性)までは設計者が事前に固定する。創発させるのは「具体例レベル」「行動と内省」「物語の展開に対するキャラの応答」のみ。
- **意図的エコーチェンバーによる奥行き付与** ── 同じ骨格でも、何を取り込んだかで奥行きが分岐する。人間の選択的注意・確証バイアスの計算論的実装。
- **気質・性格層と規範層の分離** ── 「実際にどういう特性を持っているか(being)」と「自分の中でこうあるべきと判断する基準は何か(ought)」を別アーキテクチャで持ち、両者のギャップから内省を生成する(Parks-Leduc et al. 2015)。
- **気質・性格層は主人公AIから隠蔽する(implicit / explicit 非対称)** ── 主人公AIは自分の気質・性格パラメータを直接参照できない。これらのパラメータは **Perceiver/Impulsive Agent**(裏方エージェント)だけが読み取れる。主人公AIに届くのは、裏方エージェントが自然言語化した「反射的な感じ方」「身体反応」「行動傾向」だけ。これは Nisbett & Wilson (1977) の「人間は自分の内的過程に直接アクセスできない」という古典的知見と、McClelland et al. (1989) の「implicit motive と explicit self-attributed motive は相関しない」という知見の計算論的実装。
- **内省は「自分の行動から自分を推測する」プロセス** ── Bem (1972) Self-Perception Theory の実装。主人公AIは自分の気質を見られないので、自分の行動履歴から「最近こういう反応が多い。自分には◯◯なところがあるのかもしれない」という形で自己を推測する。これが日記に「自己発見のアーク」を自然に生む。
- **内省には必ず過去記録との統合と薄れた記憶の再解釈を含める** ── 今日の行動と感情を、自伝的エピソード層・短期記憶(key memory + 通常領域)の既存の記録と接続する工程を必ず経る。加えて、すでに記憶として薄れた過去の出来事に対して、今日の経験を通じた再解釈を試みる(「あの時のあれは、こういうことだったのかもしれない」)。これがないと内省が「今日の出来事を今日の基準で評価するだけ」になり、物語的厚みが出ない。
- **感情は状態ではなくフィルター** ── 感情は知覚を変える装置。同じ出来事でも気分で見え方が変わる。持続するのは **ムード(PAD 3次元)だけ**、具体的感情(喜び・悲しみ・怒り等)は瞬間的に生じてログに残り消える。
- **3時制の内面** ── 過去志向(記憶)、現在志向(知覚)、未来志向(理想自己)を全部持つ。
- **内部はfloat、外は自然言語** ── パラメータの内部表現は数値でいいが、LLMに渡すときは自然言語に翻訳する。
- **動的活性化方式** ── 全パラメータは保持しつつ、シーンごとに関連するものだけを抽出して発火させる。気質・性格層・規範層の両方に共通するルール。
- **静的ランキングは持たない** ── 価値観は1〜19位の順位で持たない。シーンごとに関連価値が活性化し、キャラ自身に動的に葛藤させる。
- **マクロ層は常に渡す、ミクロ層は動的に活性化する** ── 名前・職業・家族構成・言語的指紋などの「キャラの常時的な輪郭」は動的活性化の対象外。常に全プロンプトに同梱する。気質・性格・規範のミクロパラメータだけが動的に活性化される。
- **自伝的エピソードは生成して保持する** ── 価値観や夢の根に必ず「経験的な物語」を持たせる。Phase A-3 で 5〜8 個の決定的エピソードを生成し、Conway の Lifetime Period 層として常時全文同梱する(retrieval は使わない)。日記の「過去への参照」はこの DB から引かれる。
- **Reflective-Impulsive Model(Strack & Deutsch 2004)を中核アーキテクチャとして採用** ── 衝動系と熟慮系は並列に動く。片方が片方を抑えるのではなく、両者の出力が競合または協調し、個人差パラメータが最終行動の重みを決める。強い情動は熟慮系の重みを下げる(Loewenstein のvisceral influences)。
- **4 類型は性格類型としては維持するが、構造分岐としては実装しない** ── 「その場の熟慮」(#13 衝動性 / #29 慎重さ)と「事後の内省」(#48 内省傾向)は別軸として性格類型(熟考型・計算型・衝動後悔型・快楽現在型)を定義する材料にはなるが、日次ループのどのステップが走るかを構造パラメータとして制御することはしない。これらは自然言語パラメータとして LLM に渡され、LLM がエージェンティックに内省の深さ・行動選択の熟慮度を調整する。整合性は **バリデーションAIが事後チェック** する。
- **情報の重要度はスコアではなく、key memory 抽出と段階圧縮で扱う** ── 数値スコアを各出来事に付与する方式は採用しない。代わりに日記生成直後に、日記生成AIが「本日最も心に残った出来事」を 1 日 1 個・300 字以内で key memory として抽出する。それ以外の出来事は段階圧縮方式で日ごとに要約度合いを深めていく。
- **省略も設計** ── キャラが何を書かないかも、人格表現の一部。複数の段階で選択的脱落を連鎖させ、結果として「書かないものの偏り」を個性として出す。
- **葛藤は偶然ではなく意図的に投入する** ── 脚本AIにキャラのプロファイルを渡し、葛藤誘発型イベントを生成させる(antagonist principle の計算論的実装)。
- **内省フェーズが価値観更新の唯一の窓口** ── 出来事から直接価値観を更新する経路は持たない。必ず内省を経由する。
- **価値観の中核は動かさない** ── 7日間で価値観の骨格(Schwartz 19 価値の強弱)が劇的に変わるのは非現実的。動くのは暗黙の優先順位と、活性化パターンの蓄積のみ。
- **主人公AI とは、日次ループ内の判断・出力系の総称である** ── 主人公AIは単一のエージェントではなく、複数のエージェント機能の集合体。詳細は §0.4 を参照。

### 0.1 キャラの能動性経路の明示化

本アーキテクチャでは日記生成後に、主人公AI 自身が「明日やりたいこと」を出力するステップ(§4.9.4 翌日予定追加エージェント Stage 1)を置く。これらの候補のうち採択されたものは、翌日のイベント列に `source: protagonist_plan` として挿入される。この経路により、「キャラが自分の意志で明日の何かを動かす」という能動性が物語構造に組み込まれる。

この経路は Generative Agents(Park et al. 2023)にも存在しない、本アーキテクチャ固有の差別化要素である。

### 0.2 既知/未知の厳密分離

脚本AIが生成するイベントには、各イベントに「主人公AIがそのイベントを事前に知っていたかどうか」を示すメタデータ `known_to_protagonist` を必須で付与する。曖昧な中間状態は許容しない。既知イベントには `source`(ルーティン由来 / 事前の約束 / 主人公の能動計画)も付く。この厳密分離は、下流の Perceiver と統合エージェントが、既知と未知で異なる質感の反応を生成できるようにするための設計である。詳細は §4.2 を参照。

### 0.3 メモリアーキテクチャ

記憶系は以下の 3 層構造で整理されている:

1. **自伝的エピソードDB(Phase A-3 で生成、7日間完全不変)** ── 5〜8 個の決定的エピソード。retrieval は使わず、全エージェントのプロンプトに全文ベタ貼りで渡す
2. **短期記憶DB(日次更新)** ── 2領域構造
   - **key memory 領域(保護領域)** ── 日記生成AIが 1 日 1 個・300 字以内で抽出する「本当に重要だった瞬間」。7 日間フル保持、書き換え不可、段階圧縮の対象外
   - **通常領域(段階圧縮)** ── 当日の行動ログ要約。日数経過に応じて日ごとに要約度合いが深まる(当日 400〜600 字 → 1日前 200〜300 字 → 2日前 80〜150 字 → 3日前以上 20〜30 字の断片のみ)
3. **日記本文ストア** ── 7 日分の日記本文、圧縮せず全文保持

記憶の重要度判定は「key memory 抽出」で、時間経過による薄れは「段階圧縮」で実装する。retrieval 機構(ベクトル検索・タグマッチング)は一切使わない。7 日間という短期スパンならコスト的に許容できるため、必要なものは全部プロンプトにベタ貼りする。

### 0.4 主人公AI の内部定義

「主人公AI」という呼称を明確化する。

**主人公AI = 日次ループ内の判断・出力系の総称**。以下のエージェント機能が主人公AIに含まれる:

- Reflective system(理性ブランチ、§4.6 Step 2)
- 行動決定エージェント(§4.6 Step 3)
- 情景描写・後日譚生成エージェント(§4.6 Step 4、v10 新設)
- 内省エージェント(§4.7)
- 日記生成エージェント(§4.8)
- key memory 抽出エージェント(§4.9.3、日記生成AIを再利用)
- 翌日予定追加エージェント Stage 1(§4.9.4、主人公AIの能動性経路)

**裏方エージェント(主人公AIではない)** = キャラ骨格層にアクセスする感情・知覚フィルター層:

- Perceiver(知覚フィルター、§4.3)
- Impulsive Agent(衝動ブランチ、§4.6 Step 1)
- 裏方エージェント出力検証エージェント(§4.6b)
- 翌日予定追加エージェント Stage 2(§4.9.4、主人公AIから隠蔽される整合性調整役)

**価値観違反チェックエージェント(§4.6c)** は主人公AI側(規範層にアクセスする判定役)に分類する。ただし機能的には「最終行動を客観的に審査する」独立専門ロールとして扱い、主人公AIの他のサブエージェントとは別の責務を持つ。

**構造上の核心:** 気質・性格層を読めるのは Perceiver / Impulsive Agent だけ。主人公AIは自分のキャラ骨格層を直接見ることはできない。この非対称性が本アーキテクチャの中核である。

## 2. キャラクター生成フェーズ(Day 0)詳細

Day 0 は 4 つの Phase で構成される。Phase A〜C はキャラ骨格と知識基盤の構築、Phase D は 7 日分のイベント列一括事前生成である。

### 2.1 Phase A:キャラ骨格の事前固定(A-1 / A-2 / A-3)

Phase A は、設計者のマクロ設定 → 脚本AI によるミクロ生成 → 脚本AI による自伝的エピソード生成、という 3 段構成で進む。

#### 2.1.1 Phase A-1:マクロプロフィール生成

設計者(または脚本AI)が、キャラの **常時的な輪郭** を一括生成する。ここで作られるマクロプロフィールは **常にプロンプトに渡され、動的活性化の対象外** となる。これは §3.1 に示す 4 層構造の最上位層に対応する。

**マクロプロフィールテンプレート:**

```
【基本情報】
  名前:[フルネーム]
  年齢:[数値]
  性別:[自由記述]
  外見の特徴:[1〜2文]

【社会的位置】
  職業:[具体的な職種、所属、肩書]
  職場・所属組織:[具体的な名前 + 規模感]
  住んでいる場所:[都市、地域、住居形態]
  経済状況:[ざっくり、生活水準を示す程度]
  社会階層・出自:[1文]

【家族・親密圏】
  家族構成:[現在同居している家族]
  両親(故人含む):[一文ずつ、関係性も]
  配偶者・パートナー:[いれば、関係性の質感]
  子供:[いれば]
  きょうだい:[いれば]
  親友・継続的な人間関係:[2〜4人、関係性の質感]

【現在の生活の輪郭】
  典型的な平日の概形:[1パラグラフ]
  典型的な週末の概形:[1パラグラフ]
  習慣・ルーティン:[3〜5個]
  趣味・余暇の使い方:[2〜3個]

【夢と目標(時系列で)】
  子供時代の夢:[何になりたかったか、なぜか]
  10代後半の夢:[同上]
  挫折・転機:[いつ、何が起きてどう変わったか]
  現在の夢・目標:[長期 / 中期 / 短期、それぞれ]
  夢の根にある何か:[1文、価値観との接続]

【話し方・文体の指紋】
  一人称:[僕 / 俺 / 私 / うち / 自分 / その他]
  二人称(状況別):[親しい人には / 目上には / 知らない人には]
  口癖:[3〜5個。実際のフレーズで]
  文末傾向:[「〜だ」「〜である」「〜だよね」等の頻度]
  漢字 vs ひらがな傾向:[硬い / 柔らかい / 揺れる]
  絵文字・記号の使用:[使う / 使わない / 限定的]
  自問形式の頻度:[よく自問する / しない]
  比喩・反語の頻度:[よく使う / 控えめ]
  避ける語彙:[「成長」「気づき」など、本人が嫌う語]

【価値観の中核(narrative 形式で書く)】
  最も大事にしていること:[1〜2文]
  絶対に許せないこと:[1〜2文]
  自分が誇りに思っていること:[1〜2文]
  自分が恥じていること:[1〜2文]

【秘密 / 公にしないこと】
  人には言わないこと:[2〜3個]
  日記にも書かないかもしれないこと:[1〜2個]

【関係性ネットワーク(登場キャラクターとの関係性)】
  人物名 | 関係 | 質感(好き/苦手/複雑) | 最後に会ったのはいつ
```

**重要な原則:**
- マクロプロフィールは **動的活性化されない**。日次ループのほぼ全ステップに常時同梱される。
- これがあることで、キャラの「言語的指紋」「秘密」「常時的な人間関係」などが、シーンごとの活性化判定で取りこぼされる事態を防げる。
- 週間固定予定・ルーティンもマクロプロフィールに埋め込む。Phase D の脚本AI はマクロプロフィール内のルーティンを参照して、既知イベントの `source: routine` を生成する。

#### 2.1.2 Phase A-2:ミクロパラメータ生成

設計者(または脚本AI)がキャラの内部パラメータを定義する。

| 層 | 項目 | 内容 |
|---|---|---|
| 気質・性格層 | 気質パラメータ(23個) | 自然言語で各値を記述 |
| 気質・性格層 | 性格パラメータ(27個) | 自然言語で各値を記述 |
| 気質・性格層 | 対他者認知層(2個) | SCO 強さ×質、嫉妬気質 感受性×質 |
| 認知パラ | 自動導出 | 学習率α、感情慣性、RPE感受性、減衰係数λ |
| 規範層 | Schwartz 19価値の強弱 | 各価値を「強・中・弱」で固定 |
| 規範層 | MFT 6+ 道徳基盤 | 骨格固定 |
| 規範層 | 理想自己 / 義務自己 | 方向性のみ固定(延長型 or 補償型 + 大枠) |
| 規範層 | 目標と夢 | 長期・中期の骨格を固定 |

**Phase A-2 規範層の原則:** Schwartz 19 価値の骨格 + 規範層の動的活性化があれば、具体例や行動方針は LLM が自然に生成できる。規範層に可変サブモジュール(明示リスト)は持たない。明示リストを持つとキャラが「ルールに従って動くロボット」的になる副作用があるため、持たない方針を採る。

**認知パラメータの自動導出ルール:**
- 学習率 α ← NS + (1-HA)
- emotional_inertia ← Persistence + HA
- RPE感受性 ← NS(ドーパミン系直接対応)
- 減衰係数 λ ← Persistence / NS の比

**性格類型のラベル付け:**

以下の 3 パラメータは、性格類型(熟考型・計算型・衝動後悔型・快楽現在型)のラベル付けに使う。日次ループの構造分岐は制御しない。これらは自然言語パラメータとして LLM に渡され、LLM がエージェンティックに振る舞いを調整する。整合性は §4.7 / §4.8 のバリデーションAI が事後チェックする。

| # | パラメータ | LLM への影響 |
|---|---|---|
| 13 | 衝動性(Impulsivity) | 行動選択時、衝動系出力の重みに影響する自然言語記述 |
| 29 | 慎重さ(Deliberation) | 行動選択時、理性系出力の生成量に影響する自然言語記述 |
| 48 | 内省傾向(Reflectiveness) | 内省フェーズの深さ・日記の内省的描写の量に影響する自然言語記述 |

#### 2.1.3 Phase A-3:自伝的エピソード生成(脚本AI)

**目的:** Phase A-1 と A-2 から、キャラの人生における **決定的なエピソードを 5〜8 個** 生成する。これにより、価値観・夢・恐れの根に「経験的な物語」が必ず存在する状態を作る。日記の中で「過去の経験への参照」が生じたとき、キャラはここから参照する。自伝的エピソードDB は retrieval を使わず、**全文ベタ貼り** で各エージェントのプロンプトに毎回渡される。

**理論的アンカー:** McAdams の物語的アイデンティティ理論(narrative identity)、および Conway の Self-Memory System における Lifetime Period 層。

**処理フロー:**

```
【Phase A-3 自伝的エピソード生成】

入力:
  ├ Phase A-1 のマクロプロフィール
  └ Phase A-2 のミクロパラメータ

処理:
  Step 1: マクロ + ミクロを脚本AIに渡す
        ↓
  Step 2: 「このキャラの人生における決定的なエピソードを 5〜8 個生成せよ」
        各エピソードに以下のメタデータを必ず付与:
          - 時期(子供時代 / 思春期 / 青年期 / 直近)
          - 種別(redemption / contamination / commitment / encounter / loss)
          - 関与した他者(マクロプロフィールの関係者から選ぶ、または新規)
          - そのエピソードが現在のどの価値観 / 目標 / 怖れに繋がっているか
          - 短い narrative(200〜400字)
        ↓
  Step 3: 各エピソードを Phase A-2 のミクロパラメータと整合性チェック
        「このエピソードは気質・性格と矛盾しないか」
        矛盾があれば再生成
        ↓
  Step 4: 自伝的エピソードDB に保存(.md ファイル、全 5〜8 個 + メタデータを1ファイルに)

制約(redemption bias 対策):
  - 少なくとも 1 個は contamination sequence(良かったものが損なわれた話)
  - 少なくとも 1 個は未解決の loss(まだ整理がついていない喪失)
  - 少なくとも 1 個は ambivalent(自分でもどう感じているか曖昧)
  - redemption sequence は最大 2 個まで

特例:夢や重要な価値観についてのエピソードは必ず含める。
  - 夢がある場合 → 夢の起源となるエピソードを必ず 1 つ含める
  - 夢がない場合 → なぜないのか、昔の夢、昔あこがれたものについてのエピソードを入れる

出力:
  自伝的エピソードDB(.md 形式、1 ファイルに全 5〜8 個)
```

**McAdams のエピソードカテゴリ(テンプレート):**

| カテゴリ | 構造 | 例 |
|---|---|---|
| Redemption sequence | 困難 → 救済 → ポジティブな現在 | 「病気で苦しんだが医師に救われた → 医者を志す」 |
| Contamination sequence | 良かったもの → 損なわれた → ネガティブな現在 | 「幸せな家庭だったが両親が離婚した → 親密な関係を怖れる」 |
| Commitment story | 誰かを助ける選択 → 意味の発見 → 使命感 | 「ボランティアで子供たちと出会った → 教師を志す」 |
| Foundational encounter | 決定的な出会い → 道が開けた | 「あの先生に出会わなければ今の自分はない」 |
| Loss & survival | 大切なものを失った → それでも進んできた | 「親友を失った → 命の有限性を意識して生きる」 |

**redemption bias の根拠:** Nature Humanities and Social Sciences Communications (2026) の研究で、LLM はデフォルトで「困難 → 救済 → 成長」のパターンに極端に偏ることが指摘されている。プロンプトで明示的に contamination, loss, ambivalence を含めることを強制しないと、5〜8 個全部が redemption sequence になりかねない。人格に陰影を出すためには、この偏りを構造的に抑制する必要がある。

### 2.2 Phase B:エコーチェンバー構築(軽量・概要DB)

骨格を持ったキャラ自身が、ネット情報をクロールして「自分にとって何が大事か」を **選択的に** 蓄積する。脚本AI が代わりに評価するのではなく、**キャラ自身の価値観プロファイルがフィルターになる**。

**位置付け:** Phase B は「概要的・嗜好的データベース」として実装する。詳細な専門知識を構築するのが目的ではなく、「この人はこういうことに興味がある」程度のフィルターをキャラに与えるのが目的。

**データソース(確定):**

| ソース | 役割 |
|---|---|
| **はてなブックマーク 人気記事** | 意見性が強く、ブクマコメント付き。感情反応を引き出しやすい |
| **note の人気記事** | 個人的体験・エッセイ系、日本語の意見性が豊富 |
| **はてなブログ** | 深い個人ブログ、長文の感想・体験 |

**処理形式:** 全文ではなく **メタデータレベル**(タイトル + 冒頭 150〜200 文字 + ハッシュタグ・ブクマコメント上位)で評価させる。

**深掘り型クロール(Curiosity-Driven Crawling)方式の採用:**

Phase B のクロールは「ランダムに記事を順次見せる」のではなく、**キャラの好奇心に従って深掘りしていく** 方式を採用する。これは強化学習における好奇心駆動型探索(Pathak et al. 2017)に対応する設計。

```
【深掘り型クロール処理フロー】

入力:
  ├ キャラの骨格プロファイル(気質・性格・価値観)
  └ 初期シードリスト(10〜20件の多様なテーマ)

処理:
  Loop(N 周、例えば 5 周):
    Step 1: 現在のキューから 1 件を取り出す
        ↓
    Step 2: その情報のメタデータをキャラに見せて評価
        ├ 好き / 強く惹かれる → 深掘り対象に決定
        ├ 嫌い / 拒否反応 → 感情的知識DBに記録、深掘りはしない
        └ どうでもいい → スキップ
        ↓
    Step 3: 「好き」「嫌い」と評価したものを感情的知識DBに記録
        ↓
    Step 4: 「好き」と評価したもののみ、深掘り処理
        ├ そのテーマから関連キーワードを 3 つ抽出
        │   (キャラに「これに関連して、もっと知りたいことを 3 つ挙げて」と聞く)
        ├ 関連キーワードで次のクロール候補を取得
        └ キューに追加
        ↓
    Step 5: ループ回数 or キュー枯渇でループ終了

出力:
  ├ 感情的知識DB(深さ方向に育っている)
  ├ クロール経路ログ
  └ 「興味の中心」マップ
```

**重要な設計判断:キーワード抽出はキャラに任せる。** Step 4 の関連キーワード抽出は機械的にやらず、キャラ自身に「次に何を知りたいか」を語らせる。これによってキャラの **好奇心の方向性自体が個性化** する。

### 2.3 Phase C:擬似経験構築(本実装では保留、概念のみ維持)

**本実装での扱い:** Phase C は本実装では作らない。設計思想としては残し、説明資料では「理想実装版では中核フェーズ、提出版は時間的制約により保留」と明記する。以下は概念記述のみ。

**目的(概念):** キャラに「経験」を持たせる。実際には経験していないが、他者の経験ストーリーを読んで「自分だったらどうするか」を擬似的にシミュレートさせ、その擬似的な経験内容を保持する。

**データソース候補(参考):** OKWave / Yahoo 知恵袋の人生相談、note の体験談記事。

**保留とした理由:**
- 時間的制約
- 本実装では自伝的エピソードDB 5〜8 個を全文同梱することで、過去経験的な厚みの代替とする

### 2.4 Day 0 の重みづけ(優先順位)

| Phase | 重要度 | 実装の厚み |
|---|---|---|
| **Phase A-1:マクロプロフィール** | **最重要(基盤)** | 設計者が手動で時間をかける、または脚本AI で生成 |
| **Phase A-2:ミクロパラメータ** | 最重要(基盤) | 設計者が手動で固定、認知パラは自動導出 |
| **Phase A-3:自伝的エピソード生成** | **最重要(基盤)** | 脚本AI で 5〜8 個、McAdams カテゴリ強制 |
| **Phase D:脚本AI による7日分イベント列一括生成** | **最重要(物語骨格)** | 1 回のみ、7 日分合計 28〜42 件 |
| Phase B:エコーチェンバー構築 | 重要(色付け) | 軽量実装、概要レベル 50〜100 件 |
| Phase C:擬似経験構築 | **保留(本実装では作らない)** | 説明資料で理想実装版として言及 |

### 2.5 Phase D:脚本AI による7日分イベント列の一括事前生成(v10 中核)

**目的:** 7 日間の物語アークを脚本AIが俯瞰した上で、7 日分 × 各日 4〜6 件(合計 28〜42 件)のイベント列を一括で事前生成する。日次ループ中の新規イベント生成は一切行わず、事前生成されたイベント列を順次注入するだけになる。

**Phase D の責務:**

1. Day 0 の最終段階として実行される(Phase A〜C の完了後)
2. 脚本AI が 7 日分 × 各日 4〜6 件 のイベント列を一括生成
3. 7 日間の起承転結アーク(Day 1=弱 → Day 5=山場 → Day 7=収束)を俯瞰した上で設計
4. 各イベントに「既知/未知 × 予想外度」の 2 軸メタデータ + 「このキャラに意味を持つ理由」を付与
5. Day 5 の山場に向かう伏線を Day 1〜4 に仕込む
6. 7 日間全体の物語連続性(前日の出来事が翌日に跳ね返る)を設計する

**なぜ一括事前生成なのか:**

- 日次生成方式では、脚本AI が 7 日全体の物語アークを俯瞰できない
- 一括生成することで、Day 5 山場への伏線を Day 1〜4 に仕込むような脚本的設計が可能になる
- 物語連続性(前日の出来事が翌日に跳ね返る構造)を設計段階で保証できる

**既知/未知の厳密分離(最重要):**

脚本AI がイベントを生成する際、各イベントに「主人公AI がそのイベントを事前に知っていたかどうか」を明示的に区別するメタデータを必須で付与する。曖昧な中間状態は許容しない。

**【既知イベント】**(`known_to_protagonist: true`)

ソースは 3 種類、脚本AI は生成時にどれかを明示する:

1. **ルーティン由来**(`source: "routine"`) ── マクロプロフィールの週間固定予定・習慣・ルーティンから展開されたもの
2. **事前の約束**(`source: "prior_appointment"`) ── 他者との事前取り決めによる予定
3. **主人公の能動計画**(`source: "protagonist_plan"`) ── 翌日予定追加エージェント(§4.9.4)を経由して後から挿入されたもの。**Day 1 には存在せず、Day 2 以降のみ出現しうる**

**【未知イベント】**(`known_to_protagonist: false`)

主人公が予見していない突発事件。他者からの予期しない連絡、偶然の遭遇、急な呼び出し、天候変化など。脚本AI がキャラに刺さるよう意図的に投入する。

**2 軸メタデータ(既知/未知 × 予想外度):**

各イベントには「既知/未知」軸に加えて「予想外度(低/中/高)」も付与する:

- **低** ── 予定通り・日常の流れ。驚き成分はほぼゼロ
- **中** ── 想定の枠内の意外性(予定はあったが議題が差し替わった、知り合いと偶然会った等)。軽い驚き・軽い揺らぎ
- **高** ── 強い驚き・動揺・価値観に関わる出来事。葛藤誘発型イベントの本拠地

この 2 軸メタデータは、Layer 1〜3 という概念的区分を自然に吸収する:

| Layer(概念) | 2 軸メタデータ | 役割 |
|---|---|---|
| Layer 1 | 既知 × 低 | 日常の反復、安心感と基調リズム |
| Layer 2 | 既知 × 中 / 未知 × 中 | 軽い揺らぎ |
| Layer 3 | 未知 × 高 / 既知 × 高 | 葛藤誘発型(Day 5 山場の本拠地) |

**脚本AI(Phase D)へのプロンプト構造:**

```
あなたはこのキャラクターの 7 日間の物語を一括で設計する脚本 AI です。
7 日分のイベント列を、物語アークを俯瞰した上で一括生成してください。

【キャラクターの情報】
  - マクロプロフィール全文(週間固定予定・ルーティンを含む)
  - 気質・性格・価値観の骨格
  - 対他者認知スタイル(SCO / 嫉妬気質)
  - 自伝的エピソード(Phase A-3 生成の 5〜8 個、全文)

【物語的条件】
  - 合計 7 日間 × 各日 4〜6 件 = 28〜42 件

    Day 4=中〜強、Day 5=強(山場)、Day 6=余韻、Day 7=収束

【各イベントに必ず付けるメタデータ】
  1. known_to_protagonist: true / false
  2. source(known_to_protagonist が true の場合のみ):
     ├ "routine"(マクロプロフィールの週間固定予定・ルーティン由来)

- 予想外度「高」は Day 5 以外では 1 日あたり最大 1 件

## 3. キャラクター内部状態の構造

### 3.1 4層構造の概観

このアーキテクチャの内部状態は **4 つの層** から成る。上位 2 層(マクロ / 自伝的エピソード)は **動的活性化の対象外** で、下位 2 層(気質・性格 / 規範)は **動的活性化の対象**。

```
【マクロ層】── 常にプロンプトに渡す、動的活性化の対象外
  ├ 基本情報
  ├ 社会的位置
  ├ 家族・親密圏
  ├ 現在の生活の輪郭(週間固定予定・ルーティン含む)
  ├ 夢の時系列
  ├ 言語的指紋
  ├ 価値観の中核(narrative 形式)
  ├ 秘密 / 公にしないこと
  └ 関係性ネットワーク

【自伝的エピソード層】── Phase A-3 で生成、全文ベタ貼り
  └ 5〜8 個のエピソード(Conway の Lifetime Period 層)
     各エピソードは現在の価値観/夢/恐れと接続済み
     retrieval を使わず、全エージェントのプロンプトに全文同梱

【気質・性格層】── 動的活性化の対象
  └ 52 パラメータ(気質 23 + 性格 27 + 対他者認知 2)

【規範層】── 動的活性化の対象
  └ Schwartz 19 価値 + MFT + Higgins(理想自己 / 義務自己) + 目標と夢
```

**層ごとのアクセス方式の差:**

| 層 | アクセス方式 | 主人公AIから直接見えるか | プロンプトへの渡し方 |
|---|---|---|---|
| マクロ層 | 常時参照 | 見える | 全ステップに同梱 |
| 自伝的エピソード層 | 常時参照(retrieval 不使用) | 全文見える | 全エージェントに全文ベタ貼り |
| 気質・性格層 | 動的活性化 | **見えない(隠蔽)** | **Perceiver/Impulsive Agent だけが読み、主人公AIには自然言語化された反射反応のみが渡される** |
| 規範層 | 動的活性化 | 見える | シーンに関連する数個だけ抽出 |

**隠蔽原則:** 気質・性格層は主人公AIから構造的に隠される。主人公AIは自分の衝動性・HA・NS といったパラメータの値を知らない。これらを読み取れるのは Perceiver/Impulsive Agent(裏方)だけで、主人公AIに届くのは「今日はなんだか胸がざわついた」「反射的に身構えてしまった」といった **自然言語化された反射反応** のみ。詳細は §3.8 に記述。

### 3.2 概念上の厳密な区別 ── 気質・性格層 vs 規範層

このアーキテクチャの核心思想は、**気質・性格層と規範層を厳密に分離する** こと。

| | 気質・性格層 | 規範層 |
|---|---|---|
| 何を表すか | その人が**実際に持っている**特性・気質 | その人が**判断・評価に使う基準** |
| 性質 | 「ある(being)」 | 「べき(ought)」 |
| 例 | 「私は怠惰である」 | 「勤勉であるべきだと思っている」 |
| 動くか | 中核は不変、表層は経験で形成 | 中核は不変、動的活性化で発火パターンが変わる |
| 出典系 | Big Five / HEXACO / Cloninger Temperament | Schwartz / MFT / Higgins |

**両者がズレることの設計上の意味:**

- 気質・性格層:#24 勤勉性 = 低(実際には怠惰)
- 規範層:Schwartz「達成」価値 = 高(勤勉であるべきと信じている)
- → ギャップが「自分は怠け者でダメだ」という内省を生む

両者を独立に持つことで、内省の質が劇的に多様化する。

#### パラメータ間の複合効果(設計例)

- **RD低 × Industriousness高** → 「他人の評価を気にせず黙々と働く」系
- **NS高 × Self-Discipline低** → 「次々手を出して終わらせない」系
- **SCO高 × 悪性嫉妬高 × Achievement高** → 「他人の成功を見て怒りで燃える」系
- **HA高 × Reflectiveness高 × Self-Acceptance低** → 「常に自分を責める内向型」系

### 3.4 規範層 ── 価値の基準

規範層は以下の要素で構成される:

| 項目 | 内容 | 可変性 |
|---|---|---|
| **B-1 価値観** | Schwartz 19 価値の強弱(骨格のみ) | **不変**(Day 0 で固定) |
| **B-2 道徳基盤** | MFT 6+(Care/Harm, Equality, Proportionality, Loyalty, Authority, Sanctity, Liberty 等) | 不変 |
| **B-3 理想自己 / 義務自己** | | |
| ├ Ideal Self 状態側面 | 「幸福でいたい」「自由でいたい」 | 微動 |
| ├ Ideal Self 振る舞い側面 | 「勇気ある人でいたい」「誠実でいたい」 | 微動 |
| └ Ought Self | 「責任を果たさねばならない」 | 微動 |
| **B-4 目標と夢** | 長期的夢 / 中期的目標 / 日常的目標 | 中期以下は可変 |

**規範層の原則:** 規範層に可変サブモジュール(価値観ごとの具体例の明示リスト、行動方針の明示リストなど)は持たない。Schwartz 19 価値の骨格 + 規範層の動的活性化があれば、シーンごとの具体的な好み・憧れの対象、および「この状況ではどう行動するか」は LLM が自然に想起・生成できる。明示リストを持つとキャラが「ルールに従って動くロボット」的になる副作用があるため、持たない方針を採る。

### 3.5 動的活性化方式

**核心原則:全パラメータは保持しつつ、シーンごとに関連するものだけを抽出して発火させる**。

これは気質・性格層と規範層の両方に適用される共通ルール。

#### なぜ動的活性化方式が必要か

- 52 個 + 規範層全部を毎回プロンプトに渡すとサイズが肥大化する
- 人間も全ての性格特性を常時意識しているわけじゃない。状況によって関連する自己側面だけが活性化する(自己複雑性理論 Linville 1985)
- どのパラメータが重要かはシーンによって変わる。静的 Tier 分けでは捉えられない

#### 処理フロー

```
【パラメータ動的活性化】

入力:
  ├ 今回のシーン(出来事の自然言語記述)
  ├ 全52パラメータ + 規範層全要素
  └ キャラの最近の感情状態(現在ムード)

処理:
  Step 1: 活性化判定AI(分類エージェント)を呼ぶ
        「このシーンに対する応答に関わってきそうな
         気質・性格・価値観・理想を抽出してください」
        ↓
  Step 2: 関連要素を可変個(目安 5〜10 個)抽出
        ↓
  Step 3: 抽出された要素の値だけを自然言語で
        本処理エージェント(知覚フィルター / 理性ブランチ / 統合エージェント)に渡す
        ↓
  Step 4: 本処理エージェントは、活性化された要素に基づいて応答を生成

出力:
  ├ 活性化された要素のログ(後で参照可能)
  └ 要素に基づいた応答
```

#### 動的活性化の含意

- プロンプトサイズ問題が解決する
- どのパラメータが発火したかがログに残るので、内省フェーズで「今日は私の中の◯◯と◯◯が反応した 1 日だった」という形で日記に書ける

### 3.6 Reflective-Impulsive Model(RIM)── 二重過程アーキテクチャ

**理論的アンカー:** Strack, F., & Deutsch, R. (2004). Reflective and impulsive determinants of social behavior. *Personality and Social Psychology Review*, 8(3), 220–247.

**中核主張:** 社会的行動は 2 つの別個の情報処理システムの作用として現れる:

- **Reflective system(熟慮系)** ── 事実と価値の知識に基づいて行動決定を生成する。**本アーキテクチャでは主人公AI 本体(理性ブランチ)が担当** し、規範層のみを参照する。主務は濃密な内面分析レポートであり、示唆と予測を明示的に含める。
- **Impulsive system(衝動系)** ── 連合的リンクと動機的指向(approach / avoidance)によって行動を引き出す。**本アーキテクチャでは裏方の Perceiver/Impulsive Agent が担当** し、気質・性格層を参照する。主人公AIはこの出力を「自然言語化された反射反応」として受け取るだけで、その背後にある気質パラメータの値は見られない。

**RIM が Kahneman の System 1 / System 2 と決定的に違う点:**

1. **両系統は常に並列で走る**
2. **両系統が同じ行動について競合または協調する**
3. **個人差(personality)が、どちらの出力が最終行動を決定するかを変える**
4. **動機的要素(approach / avoidance)が impulsive system に内在している**

#### 強い情動が熟慮系を抑制する ── Loewenstein のvisceral influences

**補完文献:** Loewenstein, G. (1996). Out of control: Visceral influences on behavior.

**中核主張:** 強い情動状態は選択肢の集合を狭める。本アーキテクチャでは Reflective 系を常に走らせ、強情動時は「重みを下げる」形で抑制を表現する(構造的スキップはしない)。

### 3.7 構造パラメータと 4 類型分岐

| # | パラメータ | 意味 |
|---|---|---|
| 13 | 衝動性(Impulsivity) | 熟考してから動く ↔ 考えずに動く |
| 29 | 慎重さ(Deliberation) | 考えずに決める ↔ 熟慮してから決める |
| 48 | 内省傾向(Reflectiveness) | 自分を振り返らない ↔ 頻繁に内省する |

**4 類型の分岐(類型の記述としては維持、実装では構造分岐しない):**

| その場の熟慮 × 事後の内省 | 結果のキャラ類型 |
|---|---|
| 高 × 高 | 熟考型・自己分析型 |
| 高 × 低 | 計算型・前進型 |
| 低 × 高 | 衝動・後悔型(文学的に最も豊か) |
| 低 × 低 | 快楽・現在型 |

**実装方針:** これら 3 パラメータは自然言語記述として LLM(Perceiver/Impulsive Agent, Reflective system, 内省フェーズ, 日記生成)に渡される。LLM がエージェンティックに振る舞いを調整する。整合性は §4.7 / §4.8 のバリデーションAI が事後チェックする。

### 3.8 主人公AIと裏方エージェント ── implicit / explicit 非対称

**理論的アンカー:**

- Nisbett, R. E., & Wilson, T. D. (1977). Telling more than we can know.
- McClelland, D. C., Koestner, R., & Weinberger, J. (1989). How do self-attributed and implicit motives differ?
- Bem, D. J. (1972). Self-perception theory.

#### 中核主張

本アーキテクチャでは、**主人公AI(日記を書く主体)と、裏方エージェント(気質・性格層を読み取って反射反応を生成する主体)を構造的に分離する**。主人公AIは自分の気質・性格パラメータを直接参照できない。

1. 主人公AIの自己認識が不完全であることが構造的に保証される
2. 内省が「行動からの自己発見」プロセスになる(Bem Self-Perception)
3. 主人公AIの自己説明が confabulate(事後でっち上げ)しうる
4. 日記生成プロンプトから気質・性格の直接記述が消える

### 4.0 現在ムードパラメータ(Current Mood State)── 日次ループの基底

**位置づけ:** 日次ループ全体を通じて参照・更新される、キャラクターの **現時点での気分(ムード)** を保持するモジュール。気質パラメータが「常に持っている性質」だとすれば、現在ムードパラメータは「今この瞬間どういう気分でいるか」を表す。両者は別物。

**保持するもの:PAD 3 次元ムード**

Russell Circumplex (1980) の 2 次元(Valence / Arousal)に、Mehrabian (1980) の PAD 3 次元モデルの Dominance を加えた 3 次元で、ムードを表現する。

| 次元 | 範囲 | 内容 |
|---|---|---|
| **Valence(快-不快)** | −5 〜 +5 の連続値 | 快の方向にあるか、不快の方向にあるか |
| **Arousal(覚醒-沈静)** | −5 〜 +5 の連続値 | 覚醒が高いか、沈静しているか |
| **Dominance(支配-無力)** | −5 〜 +5 の連続値 | 状況をコントロールできている感じか、無力感に飲まれているか |

**Dominance が加わる理由:** Valence/Arousal だけでは「不安で萎縮している状態」と「怒りで攻撃的になっている状態」の区別がつかない。両者は Valence も Arousal も同程度(ネガティブ + 高覚醒)だが、Dominance が決定的に違う(前者は無力、後者は支配的)。

**3 次元ムード空間上の典型的な位置(参考):**

| 位置 | V | A | D | 対応する気分の例 |
|---|---|---|---|---|
| 高揚・自信 | +3 | +3 | +3 | 達成感の余韻、自分が世界を動かしている感じ |
| 穏やかな満足 | +3 | −2 | +1 | リラックスした幸福感 |
| 怒り・攻撃的 | −3 | +3 | +3 | 支配的な敵意 |
| 不安・萎縮 | −3 | +3 | −3 | 逃げたい、縮こまる |
| 抑うつ | −3 | −3 | −3 | 無気力、何もしたくない |
| 退屈 | −1 | −3 | 0 | 沈滞した時間感覚 |
| 驚き(中立) | 0 | +4 | 0 | 予想外だが判断保留 |

**感情カテゴリの「参考知識」としての位置付け:**

Panksepp 7 基本情動 / OCC prospect-based 感情 / Higgins 自己評価感情 / 対他者感情(嫉妬・羞恥等)のカテゴリリストは、パラメータとしては保持しない。代わりに **各エージェントのシステムプロンプトの参考知識セクション** として渡す。LLM が自然言語で感情描写を書く際の語彙供給として機能する。

#### 処理上の役割フローチャート

```
【現在ムードパラメータの参照・更新フロー】

日の始まり(Day N 開始時):
  Step 0: 昨日(Day N-1)の終わりに carry-over された PAD 値をロード
        ↓
日次ループ全体を通じて参照される箇所:
  ├ ② 知覚フィルター(§4.3)
  │    現在ムードが知覚を歪める
  ├ ④ RIM 並列処理(§4.6 Step 1/2)
  │    現在ムードが活性化する価値観・感情描写に影響
  ├ ⑥ 行動決定エージェント(§4.6 Step 3)
  │    ムードを参照しつつ、更新後のムードも出力する
  ├ ⑦ 情景描写・後日譚生成エージェント(§4.6 Step 4)
  ├ ⑩ 内省フェーズ(§4.7)
  └ ⑪ 日記生成プロンプト(§4.8)

  Step 1: PAD 各次元への増減量を推定
  Step 2: 気質で変調(NS / HA / Persistence)
  Step 3: 更新後の PAD を確定(−5 〜 +5 にクリップ)

    反映されない。翌日の carry-over を通じてのみ効く。

  Step 1: PAD 各次元に減衰係数 λ を適用
  Step 2: |値| < 閾値 なら 0 にリセット、そうでなければ翌日に持ち越し
  Step 3: 翌日(Day N+1)の開始時の入力として保存
```

#### 気質との関係

- 気質パラメータ(Cloninger NS / HA / RD / Persistence)は **ムードの挙動を変調する**
- HA 高キャラは Valence が強く長く下がる
- NS 高キャラは Arousal + Valence が強く跳ね上がる
- Dominance は自己志向性 #44・自己受容 #45 に影響を受ける
- 減衰係数 λ は気質から自動導出される

#### 重要な原則

- **ムードはパラメータ、具体的感情はログ** ── パラメータとして持ち続けるのは PAD 3 次元のみ。具体的感情はイベントごとに LLM が描写してイベントログに記録されるが、次のイベント時には持ち越されない
- **ねじれた感情は LLM の描写で保持** ── 「親友の成功を祝福したいのに胸が苦しい」のような相反する感情の同居は、各エージェントの自然言語描写で表現する
- **carry-over の存在が最重要** ── これがないと毎日リセットされた「のっぺりした日記」になる

各イベントの `known_to_protagonist` フラグと `source` は、以下の下流処理で使われる:

- **Perceiver(§4.3)** ── 既知イベントでは「朝から気になっていた面談」のような前兆感情を生成できる。未知イベントでは純粋な驚きの源泉になる
- **理性ブランチ(§4.6 Step 2)** ── 既知イベントには事前の心構えがあり、未知イベントには無い。内面分析の組み立て方が変わる
- **統合エージェント(§4.6 Step 3)** ── `source: protagonist_plan` のイベント到来時には、プロンプトに「前日あなたは日記の中で『◯◯をする』と計画しました。実際にやるかやらないか、どのようにやるかは、今のあなたの気分・状況・優先順位で判断してください」という一文を必ず追加する
- **情景描写・後日譚生成エージェント(§4.6 Step 4)** ── 既知/未知の質感に応じて周辺描写の粒度が変わる

### 4.3 知覚フィルター(Perceiver Agent)

知覚フィルターは **独立した裏方エージェント(Perceiver Agent)** として実装する。Perceiver は気質・性格層にアクセスできるが、主人公AIはできない。Perceiver の出力は、気質・性格パラメータの値そのものではなく、**自然言語化された知覚内容** として主人公AIに渡される。

#### Perceiver の位置付け

- **入力側:** イベント(週間イベントストアから)、動的活性化で選ばれた気質・性格パラメータ 5〜10 個、現在ムード(PAD)、マクロ層、イベントメタデータ(既知/未知・source・予想外度)
- **出力側:** 自然言語の知覚ログ(後述の 3 要素)、主人公AIに渡される
- **アクセス権:** 気質・性格層(23 + 27 + 2 = 52 パラメータ)への読み取り権を持つ唯一のエージェント

#### Perceiver の出力仕様

Perceiver が出すのは以下の **3 要素のみ**。価値判断・規範照合・意味解釈は一切含まない。

1. **現象的記述(Phenomenal description)** ── 感覚モダリティに忠実な出来事の描写
2. **反射的感情反応(Reflexive affective response)** ── 身体感覚レベルの情動
3. **自動的注意配分(Automatic attention allocation)** ── 何に目が行ったか・何が視界から消えたか

#### Perceiver が出してはいけないもの

- 価値判断(「自分が悪い」「上司はひどい」)
- 原因帰属(「なぜそうなったか」)
- 行動意思決定(「どうすべきか」)
- 自己特性の言語化(「自分は怒りっぽい」)
- パラメータへの直接言及(「HA が高いので」「感情パラメータ #5 が発火した」)
- 自己特性からの帰属(「私の性格から、こう感じた」)

#### Perceiver プロンプト構造

```
あなたはこのキャラクターの「裏方の知覚エージェント」です。
キャラ本人には見えない気質・性格パラメータを読み取り、
それに基づいて「今このキャラが知覚した内容」を生成してください。

【マクロ層】(常時同梱)
  ...

【活性化された気質・性格パラメータ】(5〜10 個)

【現在ムード(PAD 3 次元)】

【イベント】
  内容:...
  known_to_protagonist: true / false
  source(known の場合):routine / prior_appointment / protagonist_plan
  予想外度: 低 / 中 / 高

【あなたのタスク】
以下の 3 要素を、「その時このキャラが何をどう感じ、
何に目が向き、身体がどう反応したか」の質感が
読み手に伝わるくらい詳しく書いてください。
省略は後段の工程でするので、ここでは描写を厚く出すことを
優先してください。

1. 現象的記述:感覚モダリティに忠実に、何が起きたか
2. 反射的感情反応:身体感覚レベルでどう感じたか(複数感情を並列で)
3. 自動的注意配分:何に目が行ったか、何が意識から消えたか

【既知イベントの場合の追加指示】
  - 事前に知っていた予定に対する前兆感情があれば含めてよい
  - source が protagonist_plan の場合、「昨日自分で決めたこと」を
    今どう感じているかを含める

【未知イベントの場合の追加指示】
  - 突発的に起こった出来事としての純粋な驚きや反応を書く

【禁止事項】(厳守)
- 価値判断をしない
- 原因を推測しない
- 行動を決めない
- 自己特性を言語化しない
- パラメータ名・番号・英字名を一切書かない
- 自己特性からの帰属を書かない
- パラメータは「仕組み」として言及するのではなく、
  「反応の質感」としてだけ出力に現れるべき
```

**Perceiver プロンプトの正のガイダンス原則:** 「描写を厚く出すことを優先」という指示により、粒度問題(厚すぎると価値判断が混入、薄すぎると具体性が消える)を「正のガイダンス + 禁止事項」の二本立てで解決する。

### 4.4 パラメータ動的活性化

§3.5 に詳述。シーンに関連する気質・性格・規範要素を抽出して発火させる。マクロ層と自伝的エピソード層は動的活性化の対象外であり、本ステップとは独立に常時参照される(マクロ層は全プロンプトに同梱、自伝的エピソード層は全文ベタ貼り)。

### 4.5 情動・理性レポート統合 + 感情更新(統合エージェントの基本方針)

本節は以下の工程の配置を概観する。詳細は §4.6 に記述する。

- **自己不一致評価(Higgins)** → 統合エージェントに配置(§4.6 Step 3 Sub-step 3-1)
- **動的価値葛藤** → 理性ブランチ内部 + 統合エージェント両方に関与(§4.6 Step 2, Step 3)
- **ムード更新** → イベント単位は統合エージェント(§4.6 Step 3 Sub-step 3-5)、日次単位は §4.9.2
- **記憶の選択的保持** → key memory 抽出ステップ(§4.9.3.1)
- **予測誤差ベースの感情** → 脚本AI のイベント生成時に付与される **予想外度タグ** と、下流 LLM の文脈的生成に委ねる(数値スコアは持たない)

**統合エージェントの入力:**

- 衝動・理性の両レポート(§4.6b 検証フェーズ通過済み)
- 理性ブランチが出した理性側ルートの予測
- イベントメタデータ(既知/未知・source・予想外度)
- 現在ムード(PAD 3 次元)
- 自伝的エピソードDB 全文
- 短期記憶DB key memory 領域 全件
- 短期記憶DB 通常領域(段階圧縮済み)
- 直近数日の日記本文
- 感情の種類 参考知識
- 理想自己 / 義務自己(Higgins 評価用)
- マクロプロフィール

詳細は §4.6 Step 3 に記述。

### 4.6 行動選択フロー(RIM 並列処理)

行動選択は **Reflective-Impulsive Model に基づく並列処理** として実装する。両系統が常に並列で走り、個人差パラメータ(#13 衝動性 / #29 慎重さ)と現在情動強度は「どちらを優先するかの重み」に影響する。

**エージェント分離:**

- **Impulsive system** ── **Perceiver/Impulsive Agent(裏方)** が担当。気質・性格層にアクセス可能。出力は自然言語化された反射反応・行動傾向として主人公AIに渡される
- **Reflective system** ── **主人公AI本体(理性ブランチ)** が担当。規範層(Schwartz / MFT / Higgins)にアクセス可能。気質・性格層は見られない

**Impulsive Agent 出力制約:**

- パラメータ名・番号・英字名を一切書かない
- 自己特性からの帰属を書かない
- 分析・原因説明を書かない
- 出力は **感情の表層と行動傾向の質感だけ** で構成する

```
【RIM 並列行動選択】

入力:
  ├ 知覚フィルター後の出来事(既知/未知・source・予想外度タグ込み)
  ├ 活性化された気質・性格・規範パラメータ
  ├ マクロ層(常時参照)
  ├ 自伝的エピソードDB 全文
  ├ 短期記憶 key memory 全件
  ├ 短期記憶 通常領域(段階圧縮済み)
  ├ 直近数日の日記本文
  ├ 現在ムード(PAD 3 次元)
  └ イベントのメタデータ

処理:
  Step 1: Impulsive system の出力生成(常に走る / 裏方エージェントが担当)
        ├ 入力:現在ムード + 気質パラメータ + 知覚された出来事
        ├ 生成内容:感情反応 + 行動傾向(approach / avoidance / freeze)
        ├ 速い、連合的、動機直結
        └ 【制約】出力にパラメータ名・自己特性への帰属・分析を
          一切含めない

  Step 2: Reflective system の出力生成(常に走る / 主人公AI本体が担当)
        = ワークフロー上の「価値観フィルター起動 + 濃密な内面分析レポート生成」
        
        【主務】
        濃密な内面分析レポートを生成する。
        「なぜそう感じるのか」「この状況は自分にとって何を意味するか」
        「価値観・知識・過去経験との接続」を濃く書き、
        示唆と予測を明示的に含める。
        
        入力:
         - 動的活性化された価値観 + 理想自己 + 義務自己
         - 過去の行動履歴、短期記憶 key memory、通常領域、日記本文
         - 自伝的エピソードDB 全文
         - Impulsive Agent の自然言語出力
         - 現在ムード(PAD 3 次元)
         - 感情の種類 参考知識(§4.0 参照)
         - イベントメタデータ(既知/未知、source、予想外度)
        
        生成内容(プロンプトに明示的に指示される):
         1. 濃密な内面分析
            「自分はなぜこう感じるのか」
            「この状況は自分にとって何を意味するのか」
            価値観・知識・過去経験との接続づけ
            「このような経験があるから」「こういう知識があるから」
            という文脈化
            感情的な自己理解の試み
         
         2. 示唆
            この状況に対してどのような方向の行動があり得るか、
            分析の中から自然に浮かび上がる示唆
            (必ずしも網羅的な選択肢列挙ではなく、
             分析の流れから出てくる示唆として書く)
         
         3. 予測(理性側ルートについて)
            自分が価値観に従ったルートを取った場合、
            起こりうる良いこと / 悪いことの予測
            (理性側ルートの予測は理性ブランチ側の責務。
             衝動側ルートの予測は統合エージェントが内部的に組み立てる)
        
        遅い、規則的、抽象的

  Step 3: 統合エージェント(行動決定)
        = ワークフロー上の「行動決定エージェント」
        
        入力:
         - 衝動・理性の両レポート(§4.6b 検証フェーズ通過済み)
         - 理性ブランチが出した理性側ルートの予測
         - イベントの既知/未知・source・予想外度メタデータ
         - 現在ムード(PAD 3 次元)
         - 自伝的エピソードDB 全文
         - 短期記憶 key memory 全件 + 通常領域
         - 直近数日の日記本文
         - 感情の種類 参考知識
         - 理想自己 / 義務自己(Higgins 評価用)
         - マクロプロフィール

        内部処理:
          Sub-step 3-1: Higgins 自己不一致評価
            Ideal 不一致(落胆系)と Ought 不一致(不安系)を
            明示的に分離して生成

          Sub-step 3-2: 予想外度タグからの予測誤差感情生成
            予想外度「高」→ 強い驚き・動揺・relief・落胆のいずれか
            予想外度「中」→ 軽い驚き・軽い肩透かし等
            予想外度「低」→ 予測誤差感情はほぼゼロ

          Sub-step 3-3: 2ルート予測の内部組み立て(β方式)
            理性側ルートの予測は理性ブランチ(Step 2)の出力に
            既に含まれている。統合エージェントはそれを参照材料として
            使いつつ、衝動側ルートの予測を内部的に組み立てる。
            最終的に両ルートの予測が並ぶ形になる:
            
            (A) 理性ブランチが推すルート(= 価値観に従った行動・選択)
                起こりうる良いこと:
                  - [理性ブランチの出力から引き継ぐ + 必要なら補完]
                起こりうる悪いこと:
                  - [理性ブランチの出力から引き継ぐ + 必要なら補完]
            (B) 衝動ブランチが推すルート(= 反射的に取りたくなった行動)
                起こりうる良いこと:
                  - [統合エージェントが内部的に組み立てる]
                起こりうる悪いこと:
                  - [統合エージェントが内部的に組み立てる]
            
            【プロンプト制約】
            - 各項目は「だいたいこうなるだろう」の語彙で書く
            - 「必ず 1 つ以上書く」の強制はしない。0 件でも可
            - 項目数は動的に変化してよい

          Sub-step 3-4: 最終行動の決定(一発出し)
            (A)(B) 両ルートの予測を並べて見比べた上で、
            以下を加味して 1 回の生成で最終行動を決める:
            ├ 衝動・理性両系統の出力の重み
            │    ├ #13 衝動性 高 + 現在ムード Arousal 高 → 衝動側に加重
            │    ├ #29 慎重さ 高 + Arousal 低 → 理性側に加重
            │    ├ 自己鍛錬 #25 / 良心性 #27 / 自己志向性 #44 高
            │    │  → 理性側に加重
            │    ├ 感情強度 #7 高 → 衝動側に加重
            │    └ Arousal 極端に高い(Loewenstein 経路)
            │      → 理性側の重みを大きく下げる(ただしスキップはしない)
            ├ Higgins 評価結果(強い不一致があれば影響大)
            └ 2ルート予測で出した良いこと・悪いこととの整合
            
            【重要】一発出し方式を採用。複数候補を生成して
            1 つを選ぶ方式(10 選択肢方式、5 候補→3 候補→1 選択)は
            採用しない。統合エージェントは 1 回の生成で最終行動を決める。
            
            【source: protagonist_plan のイベントに対する追加指示】
            主人公AI へのプロンプトに必ず以下の一文を追加する:
            「前日あなたは日記の中で『◯◯をする』と計画しました。
             実際にやるかやらないか、どのようにやるかは、今のあなたの
             気分・状況・優先順位で判断してください。」

          Sub-step 3-5: 現在ムードパラメータ(PAD)の更新
            このイベントによる PAD 各次元の増減量を推定し、
            更新後の PAD を出力する(§4.0 「イベント単位の更新」参照)

          Sub-step 3-6: 気持ちがどう変わったかの短文記述
            イベント前後の気分変化を短文で記述
            (内省フェーズと日記生成のための素材)

        出力:
         - 最終行動(1 つ)
         - Higgins 評価結果
         - 2ルート予測の並列表示
         - 更新後のムード(PAD)
         - 気持ちの変化の短文記述

  Step 4: 情景描写・後日譚生成エージェント(v10 新設)
        = ワークフロー上の「情景描写・後日譚生成エージェント」
        
        位置付け:
          行動決定エージェント(Step 3)の直後に置く。
          Step 3 で「何をしたか」まで決まっているので、本エージェントは
          その行動を受けて、具体的な情景描写とその後に起こった
          周辺的出来事を付与する役割を持つ。
        
        入力:
         - 行動決定エージェントの出力(最終行動、最終行動の短文記述)
         - Perceiver 出力(現象的記述 + 反射感情 + 自動注意)
         - 理性ブランチの内面分析レポート
         - 現在ムード(PAD、更新後)
         - イベントメタデータ(既知/未知、source、予想外度)
         - マクロプロフィール(場所・関係者など)
         - 直近数日の日記本文(文体の一貫性のため)
        
        処理:
          Step 4-1: 濃密な情景描写の生成
            - 場の空気
            - 周囲の反応(同席者の表情、声、仕草)
            - 会話の具体(誰が何と言ったか)
            - 物理的環境(光、音、匂い、温度など)
            - キャラ自身の外的な振る舞い(声のトーン、身体の動き)
          
          Step 4-2: 後日譚(その行動の後に起こった周辺的出来事)の生成
            - キャラの行動に対する他者の応答
            - 状況がどう展開したか
            - 直後の余波(余韻、気まずさ、安堵など外的な変化)
            - 次のイベントまでの間に起こった小さな出来事
              (次の大きなイベントに繋がらない程度の背景的描写)
          
          Step 4-3: 既知/未知メタデータに応じた調整
            - 既知イベントでは「予定通りだったか、予定がどう逸れたか」の
              描写に重点
            - 未知イベントでは「突然性と、その後の立て直し」の描写に重点
            - source: protagonist_plan では「自分が決めたことを
              実際にやってみてどうだったか」の描写に重点
        
        出力:
          1 つのイベントパッケージに統合される情景描写と後日譚のテキスト

  Step 5: 行動履歴バッファへの記録(§4.6c 価値観違反チェック通過後)
        記録項目は §4.6d に詳述
```

**行動履歴バッファ:**

```
【行動履歴バッファ】
  保持期間:1 日のみ
  内容:その日キャラが取った行動・決断の RAW ログ(RIM 両系統の出力込み)
  記録項目:
    - impulsive 出力 vs reflective 出力の乖離
    - 理性ブランチの内面分析レポート(全文)
    - 理性側ルートの予測(理性ブランチが出したもの)
    - 衝動側ルートの予測(統合エージェントが内部的に組み立てたもの)
    - 統合エージェントが決定した最終行動
    - 情景描写・後日譚生成エージェントの出力
    - 両系統の重み比
    - Higgins 評価結果(Ideal 不一致 / Ought 不一致)
    - 価値観違反チェックエージェント(§4.6c)の結果
    - イベント前後のムード(PAD 3 次元)の推移
    - イベントの既知/未知・source・予想外度タグ
  使用先:夜の内省フェーズで参照される
  リセットタイミング:§4.9.5 DB 更新時
```

### 4.6b 裏方エージェント出力の検証フェーズ

**位置づけ:** Perceiver(§4.3)と Impulsive Agent(§4.6 Step 1)の出力が、隠蔽原則・禁止事項を守っているかを検査する独立工程。裏方エージェントが出力した直後、その出力が主人公AIに渡される前に介在する。

#### パイプライン形式:MD ファイルベース

すべての裏方エージェントの出力は **.md ファイル** として書き出される。検証フェーズは .md ファイルを読み、編集し、最終版を確定する。

#### 検証エージェントの道具箱

| ツール | 用途 |
|---|---|
| `str_replace(target, replacement)` | 局所的な削除・置換 |
| `rewrite_sentence(target_sentence, new_sentence)` | 1 文単位の書き直し |
| `request_full_regenerate()` | 最終手段。裏方エージェント本体に再実行を依頼 |

#### 検証フェーズ処理フロー

```
【検証フェーズ】

入力:
  ├ 裏方エージェントの出力 .md ファイル

処理:
  Loop(最大 3 周):
    Step 1: .md ファイルを読む
    Step 2: 禁止事項違反を検出
    Step 3: 違反が 1 つ以上ある場合、検証AIがその場で道具を選ぶ
    Step 4: 修正を適用し、修正回数カウンタを +1
    Step 5: 再検査

        └ 違反が残っているが修正回数 < 3 → ループ継続

  ループ終了条件:
    ├ 違反なしで終了 → .md ファイル確定
    └ 3 周しても違反が残る → Opus エスカレーション

出力:
  ├ 検証済み .md ファイル
  ├ 修正ログ
  └ エスカレーションフラグ
```

**モデル選択:** Sonnet デフォルト + Opus エスカレーション。3 周失敗時に Opus に回し、Opus でも違反が残る場合は `request_full_regenerate()` で裏方エージェント本体を再実行する。

### 4.7 内省フェーズ(内省文章出力エージェント)── 日記生成の直前に配置

日記生成の **前** に置く独立工程。学習フェーズと表現フェーズを物理的に分離する。

**内省フェーズで必須の 3 工程:**

1. **自己推測(Self-Perception)** ── 主人公AIは自分の気質・性格層を見られないため、自分の行動履歴から自分の特性を推測する。Bem (1972) Self-Perception Theory の実装
2. **過去記録との統合** ── 今日の行動と感情を、自伝的エピソード層・短期記憶(key memory + 通常領域)との接続関係で見直す
3. **薄れた記憶の再解釈** ── すでに記憶として薄れた過去の出来事を、今日の経験を通じて再解釈する

この 3 工程は **キャラの #48 値に関わらず、常に内省プロンプトの中で言及される**(ただし深さは #48 に応じて変わる)。

#### 内省フェーズの内部処理

```
【内省フェーズ Internal Reflection】

入力:
  ├ マクロプロフィール(常時参照)
  ├ 今日の行動履歴バッファ(RAW、RIM 両系統の出力込み)
  ├ 価値観プロファイル(Schwartz 19 価値の骨格のみ)
  ├ 理想自己 / 義務自己
  ├ #48 内省傾向パラメータ(自然言語記述で渡す)
  ├ 自伝的エピソードDB 全文
  ├ 短期記憶DB key memory 領域 全件(過去日分含む)
  ├ 短期記憶DB 通常領域(段階圧縮済み)
  ├ 過去数日の日記本文
  └ 現在ムード(PAD)

LLM エージェンティック内省:
  - 内省 LLM は上記入力を受け取り、#48 値に応じて内省の深さ・含有率を
    エージェンティックに調整する
  - プロンプトには以下の 3 工程を必ず含めるよう指示する:

  工程 A:自己推測(Self-Perception)
    - 今日の行動履歴 + 直近数日の行動履歴を横並びで眺める
    - 反応パターンで気づくことがあれば「〜なところがあるのかもしれない」
      の形で暫定推測を書く
    - 主人公AIは自分の気質を直接見られないため、これが「自分を知る」唯一の経路
    - 推測はあくまで暫定的なもので、確定的な自己理解ではない

  工程 B:過去記録との統合
    - 自伝的エピソード層との照合:「今日の出来事は、あの時の経験に似ている」
    - 短期記憶DB key memory 領域との照合:「数日前のあの瞬間が、今日また戻ってきた」
    - 短期記憶DB 通常領域との照合:「昨日も同じような反応をしていた」
    - 過去の日記本文との照合:「昨日の日記で書いた気持ちが、今日に繋がっている」

  工程 C:薄れた記憶の再解釈
    - 自伝的エピソード + 短期記憶の中から、現在の自己目標・感情と関連するが
      薄れている記憶を 0〜2 件選ぶ
    - 今日の経験を通じて、それらを再解釈する
    - 「あの時のあれは、こういうことだったのかもしれない」
    - 再解釈結果は内省メモの一部として出力されるだけで、
      内省フェーズから記憶系DB への直接書き込みは一切行わない
    - 自伝的エピソードDB は一切更新しない(Day 0 固定)
    - 短期記憶DB(key memory 領域・通常領域)への反映経路は 2 つのみ:
      (a) 内省メモが日記生成プロンプトに入り、日記本文に反映される
      (b) 日記生成後の §4.9.3.1 key memory 抽出エージェントが
          「本当に重要だった瞬間」として内省の再解釈内容を選ぶ可能性がある
    - いずれの経路も内省フェーズ自身が書き込むわけではない

  深さ調整:
    - #48 高 → 3 工程を深く、長く、複数の行動を横断的に分析
    - #48 中 → 3 工程を標準的な深さで実施
    - #48 低 → 3 工程を浅く、短く、1〜2 件の行動についてだけ触れる
    - #48 極低 → 3 工程の言及は最小限、日常の流れを確認する程度

出力:
  内省メモ(200〜400 字程度、#48 値に応じて長さは変動)
  - 自己推測
  - 過去接続
  - (該当があれば)薄れた記憶の再解釈

バリデーションAI による事後チェック:
  - 生成された内省メモが、キャラの #48 値と整合しているかを検査
  - 不整合の例:
    ├ #48 低キャラの内省メモが、内省的な文で溢れている
    ├ #48 高キャラの内省メモが、薄い振り返りで終わっている
    ├ 3 工程のいずれかが完全に欠けている
    └ パラメータ名・番号が内省メモに混入している(隠蔽原則違反)
  - 不整合があれば修正指示を出す
  - バリデーションAI は §4.6b と同じ 3 ツール体制を使う
  - 3 周失敗時に Opus エスカレーション

価値観更新の経路:
  - 価値観本体の更新は累積閾値超えの場合のみ(慎重判断、稀)
  - それ以外の内省結果は日記の材料として使われるだけ
```

**重要な原則:**

- 出来事から直接価値観を更新する経路は存在しない。必ず内省を経由する
- 価値観本体の更新は累積閾値超えの場合のみ
- 自伝的エピソードDB は 7 日間一切更新しない。内省で再解釈した結果は短期記憶側に蓄積される
- 自己推測は、主人公AIが気質・性格層を見られないという隠蔽原則の帰結として成立する
- 過去記録との統合と再解釈は、日記の物語的厚みを生む中核工程

対応するワークフロー上のエージェント:**日記生成エージェント**

- **内省の結論を必ずプロンプトに含める**(内省と日記が乖離するのを防ぐため)
- **「省略の指示」を必ず入れる**

**プロンプト構造:**
```
【マクロプロフィール(常時同梱)】
  基本情報・社会的位置・家族・生活の輪郭(週間固定予定・ルーティンを含む)
  言語的指紋(一人称・口癖・文末・漢字/ひらがな傾向 等)

【今日起きたこと + あなたが感じたこと】
  本日のイベントパッケージ群(各イベントについて以下を同梱):
    - Perceiver 出力(現象的記述 + 反射感情 + 自動注意)
    - 理性ブランチの内面分析レポート
    - 行動決定エージェントの最終行動
    - 情景描写・後日譚生成エージェントの出力
    - 各イベントの既知/未知・source・予想外度メタデータ
    - Higgins 評価結果(Ideal 不一致 / Ought 不一致、該当があれば)

### 4.9 日記生成後の処理(日次処理)

日記生成後の処理を以下の 5 ステップに整理する。いずれも **1 日に 1 回** 実行される(外層ループ):

1. **日記出力チェック(Self-Critic、§4.9.1)**
2. **ムードパラメータ更新(1 日単位、§4.9.2)**
3. **key memory 抽出 + 短期記憶段階圧縮(§4.9.3)**
4. **翌日予定追加エージェント(§4.9.4)**
5. **DB 更新 + バッファリセット + ムード carry-over(§4.9.5)**

※ 価値観違反チェックは §4.6c に分離(イベントループ内の独立エージェント)

#### 4.9.1 日記出力チェック(Self-Critic)

対応するワークフロー上のエージェント:**日記出力チェック**

- AI 臭さ・一貫性チェック、不合格なら日記再生成
- バリデーション観点:
  - 言語的指紋(一人称・口癖・文末)が守られているか
  - キャラの #48 内省傾向に整合する内省含有率か
  - 自己特性の断定表現(「自分は◯◯な人間だ」)が混入していないか
  - パラメータ名・番号・英字名が混入していないか(隠蔽原則違反)
  - 過去日の日記・内省履歴との矛盾がないか

#### 4.9.2 ムードパラメータ更新(1 日単位)

対応するワークフロー上のエージェント:**ムード更新エージェント(1 日単位)**

**Peak-End Rule(Kahneman)+ 日記本文整合補正。**

1 日の終わりに、本日のムード(PAD 3 次元)推移を俯瞰し、1 日を締めくくる最終ムードを算出する。この最終ムードが §4.9.5 の carry-over 処理の入力となる。

※ §4.9.2 の日次集約ムード(Peak-End Rule)は当日の日記生成には反映されない。翌日の carry-over を通じてのみ効く。

**Peak-End Rule の理論的根拠:** Kahneman (1993, 2000) の実証研究が示すとおり、人は 1 日を振り返るとき、ピーク時(最大強度の瞬間)と終端時(最後の時点)の感情で 1 日全体を評価する。

**処理フロー:**

```
【ムードパラメータ更新(1 日単位)】

入力:
  ├ 本日の全イベントの PAD 推移ログ(イベントごとの V/A/D)
  ├ 本日生成された日記本文
  ├ 最終イベント終了時点の PAD(イベント単位更新の最終値)
  ├ キャラのマクロプロフィール
  └ キャラの気質プロファイル(NS/HA/RD/Persistence)

処理:
  Step 1: 1 日の PAD アークを俯瞰
        → 各次元(V / A / D)についてピーク時を特定
        → 各次元について終端時を特定

  Step 2: Peak-End Rule に基づく補正
        暫定最終値 = α × ピーク値 + β × 終端値
        ただし α + β = 1、気質で変調:
          ├ Persistence 高 → α 高め(ピーク寄り、引きずるタイプ)
          ├ Persistence 低 → β 高め(終端寄り、切り替えが速いタイプ)
          ├ NS 高 → β 高め(終端寄り)
          └ HA 高 → α 高め(ピーク寄り、特にネガティブピークに引きずられる)

  Step 3: 日記本文からの気分抽出
        LLM に日記本文を読ませ、PAD 各次元について −5 〜 +5 の値を推定

  Step 4: 日記整合補正
        暫定最終値と日記抽出ムードを比較、大きくズレている場合は
        日記抽出ムードの方向に軽く補正(最大 ±1.0)
        気質で変調:
          ├ #48 高 → 日記との整合性を強めに補正
          └ #48 低 → 日記との整合性を緩めに補正

  Step 5: 最終ムードを確定(−5 〜 +5 にクリップ)

出力:
  1 日の締めくくりの最終ムードパラメータ(V, A, D)
  補正ログ(説明資料用)
```

#### 4.9.3 key memory 抽出 + 短期記憶段階圧縮

対応するワークフロー上のエージェント:**key memory 抽出エージェント + 記憶圧縮エージェント**

本節は以下の 2 工程で構成される:

1. **key memory 抽出** ── 日記生成AI が 1 日 1 個・300 字以内で「本当に重要だった出来事」を抽出し、短期記憶DB の key memory 領域に保存する(§4.9.3.1)
2. **短期記憶 通常領域の段階圧縮** ── 記憶圧縮エージェントが当日の行動ログを要約し、過去日の要約をそれぞれ次の段階に繰り下げる(§4.9.3.2)

**実行順序の前提:**

§4.9 冒頭の 5 ステップリストの順序どおり、本節(§4.9.3)は §4.9.2 ムード更新の **後** に実行される。つまり §4.9.3.1 key memory 抽出エージェントと §4.9.3.2 記憶圧縮エージェントが参照する「現在ムード」は、いずれも **Peak-End Rule + 日記整合補正適用後の日次集約ムード** である。イベント単位更新の最終値ではない。

##### 4.9.3.1 key memory 抽出

**仕様:**

- 担当:**日記生成AI(§4.8 のエージェントを再利用)**
- 保存先:**短期記憶DB の key memory 領域**(短期記憶DB 内部の保護領域)
- 字数:300 字以内の 1 つの文章
- 件数:1 日につき必ず 1 個(0 件不可、複数不可)
- 段階圧縮対象外:7 日間フル保持、書き換え不可
- 蓄積:7 日で最終 7 個
- なぜ日記生成AI に抽出させるか:日記を書き終えた直後で、どの出来事を厚く書いたかを一番知っているから
- 参照する「現在ムード」:**§4.9.2 で算出された日次集約ムード(Peak-End Rule + 日記整合補正適用後)**。イベント単位更新の最終値ではない

**プロンプト構造:**

```
あなたは先ほどこのキャラの日記を書いた主体です。
同じ視点で、以下のタスクを追加で行ってください。

【タスク】
この 1 日の中で、あなたにとって本当に重要だった出来事、
あるいは一番心に残った出来事を 1 つだけ選んでください。
日記に書いたことでも書かなかったことでも構いません。
選んだ出来事について、なぜそれが重要だったのかも含めて、
300 字以内の 1 つの文章として書いてください。

【入力】
  - 本日の日記本文
  - 本日の全イベントパッケージ(情景描写・後日譚込み)
  - 本日の行動履歴バッファ
  - 内省メモ
  - マクロプロフィール
  - 現在ムード

【出力制約】
  - 必ず 1 個(0 件不可、複数不可)
  - 300 字以内の 1 つの文章
  - 字数オーバー時は追加指示で再試行する運用とする
```

**字数制約の扱い:** 字数オーバー時の再生成ロジックは実装者判断。本仕様書レベルでは「300 字以内、1 日 1 個」だけを規定する。

##### 4.9.3.2 短期記憶 通常領域の段階圧縮(記憶圧縮エージェント)

**担当:記憶圧縮エージェント**(§4.9.3.1 の key memory 抽出エージェントとは別立ての独立エージェント)

**仕様:**

短期記憶通常領域は「過去日ごとに要約度合いが違う段階圧縮方式」で保持する。毎日の終わりに、過去日の要約がそれぞれ次の段階に繰り下がる。

| 経過日数 | 保持形式 | 想定字数 |
|---|---|---|
| 当日(Day N) | 行動履歴バッファから 1 回要約 | 400〜600 字 |
| 1 日前 | 前日要約をさらに要約 | 200〜300 字 |
| 2 日前 | もう一段要約 | 80〜150 字 |
| 3 日前以上 | 断片のみ(key memory 以外は痕跡化) | 20〜30 字 |

**重要な原則:**

- 再要約は毎日行う
- 短期記憶DB の key memory 領域は段階圧縮の対象外(完全な形で残る)
- 日記本文は圧縮しない(7 日分すべて原形保持)
- 自伝的エピソードDB は一切更新しない(Day 0 固定)
- 参照する「現在ムード」は、**§4.9.2 で算出された日次集約ムード(Peak-End Rule + 日記整合補正適用後)**

**初期日(Day N-k が存在しない場合)のスキップガイド:**

- Day 1 時点では「昨日」「2 日前」「3 日前以上」は一切存在しない → Step 1(当日要約)のみ実行、Step 2 以降はスキップ
- Day 2 時点では「2 日前」「3 日前以上」が存在しない → Step 1 + Step 2 のみ実行、Step 3 以降はスキップ
- Day 3 時点では「3 日前以上」が存在しない → Step 1〜3 を実行、Step 4 はスキップ
- Day 4 以降は全ステップ実行可能
- 一般規則:入力として存在しない段の処理はスキップし、存在する段のみを次の段階に繰り下げる

**処理フロー:**

```
【記憶圧縮エージェントによる短期記憶 通常領域の段階圧縮】

入力:
  ├ 本日の行動履歴バッファ(全イベントパッケージ)
  ├ 本日の日記本文
  ├ 昨日の通常領域要約(Day N-1 時点で 1 日前ポジションにあったもの)
  │    ※ Day 1 では存在しない
  ├ 2 日前の通常領域要約
  │    ※ Day 1, Day 2 では存在しない
  ├ 3 日前以上の断片
  │    ※ Day 1, Day 2, Day 3 では存在しない
  ├ 日次集約ムード(§4.9.2 出力、Peak-End Rule + 日記整合補正後)
  └ キャラの特性・感性情報(要約の粒度調整用)

処理:
  Step 1: 当日(Day N)の行動ログを 400〜600 字に要約
        圧縮時のプロンプトには、キャラの特性・感性 + 日次集約ムードを注入:
        「このキャラの特性と、今日の 1 日を終えたときの気分を考慮すると、
         以下の出来事のうち、特にどの点が印象に残るかを踏まえて
         要約してください」

  Step 2: 昨日(1 日前)の要約をさらに要約 → 200〜300 字
        入力が存在しない場合(Day 1)はスキップ

  Step 3: 一昨日(2 日前)の要約をもう一段要約 → 80〜150 字
        入力が存在しない場合(Day 1, Day 2)はスキップ

  Step 4: 3 日前以上の要約を断片化 → 20〜30 字
        入力が存在しない場合(Day 1, Day 2, Day 3)はスキップ

出力:
  更新された短期記憶DB 通常領域(各日の要約を含む)
```

#### 4.9.4 翌日予定追加エージェント(v10 新設)

**位置づけ:** 日記生成後、§4.9.2 ムード更新 → §4.9.3 key memory 抽出 + 記憶圧縮 が完了した後、§4.9.5 DB 更新 + ムード carry-over の前に実行される。主人公AI 自身が「明日やりたいこと」を出力し、それを翌日のイベント列に挿入する独立工程。2 段構成で実装する。

##### Stage 1:主人公AI「やりたいこと」出力エージェント

- **担当:** 主人公AI(日記生成AI を再利用、または同等モデル)
- **入力:** 本日の日記本文、内省メモ、本日の key memory、現在ムード、マクロプロフィール、規範層、翌日の既存イベント列(概要のみ)

**プロンプト:**

```
今日の日記と内省を踏まえて、明日やりたいこと、
あるいは明日やるべきだと思うことがあれば書いてください。

もし 1 つ以上思いつくものがある場合、各項目について以下を書いてください:
  - 何をするか(1〜2 文)
  - いつ頃やりたいか(朝 / 午前 / 昼 / 午後 / 夕方 / 夜 / 深夜 の
    いずれか)
  - なぜそれをしたいのか(今日の出来事や気持ちとの繋がりを 1 文で)

もし明日やりたいこと・やるべきことが思いつかない場合、
あるいは今はそういう気分ではない場合は、そのことを明示的に書いてください。
例:
  「明日について特に今考えたいことはない」
  「今は何かを決める気分じゃない」
  「今日のことで手一杯で明日のことまで考えられない」
など、自分の今の状態として素直に書いてください。
無理に何かを挙げる必要はありません。

制約:
  - 項目は最大 3 つまで(0 個〜3 個の範囲で、今の気分に素直に)
  - 思いつくものがあれば積極的に書いてください。ただし無理に数を
    揃える必要はありません
  - 実現可能性は一旦置いておいて、「やりたい」と思ったものを
    素直に書いてください
  - 今日の日記で書いたことと矛盾しないようにしてください
```

- **出力:** 0〜3 個の候補、または「今は何も考えたくない」等の状態表明

##### Stage 2:予定挿入・整合性調整エージェント

- **担当:** 裏方の簡易整合性AI(主人公AI から隠蔽される)
- **入力:**
  - Stage 1 の 0〜3 候補(または「今は何も考えたくない」の状態表明)
  - 翌日の既存イベント列
  - マクロプロフィール
  - **性格パラメータ注入AI からの自然言語化された情報**(特に #48 内省傾向)

**性格パラメータ注入AI の仕組み:**

Stage 2 は裏方エージェントであるため、気質・性格層への読み取り権を持つ。ただし Stage 2 自身が直接パラメータ値を解釈するのではなく、§3.5 の動的活性化方式と同じ仕組みで、裏方の **性格パラメータ注入AI** が必要なパラメータを読み取り、Stage 2 に自然言語化して注入する。

注入される情報の例(#48 内省傾向が低いキャラの場合):

```
【性格パラメータ注入AI からの情報】
このキャラは内省傾向が低い傾向がある。
計画を立てる際も、あまり深く考えずに勢いで決めることが多い。
立てた計画を実際に実行に移す意志力も、強い方ではない。
そのため、高い意志力や持続的な努力を伴う計画を連発しても、
実行可能性は低い。
```

この情報を受けた Stage 2 は、Stage 1 の候補の採択判定時にこれを参考にする。ただしこの情報は **参考** であって、機械的な採択率の操作ではない。Stage 2 は LLM として自然言語の情報に基づいて総合判断を行う。

**処理:**

```
Step 1: Stage 1 の出力を確認
  ├ 0 個(状態表明のみ)の場合 → 採択処理スキップ、状態表明を
  │   翌日ログの参照用メモとして保持し、Step 3 へ
  └ 1〜3 個の候補がある場合 → 各候補について採択判定

Step 2: 1〜3 候補それぞれについて採択判定
  ├ 物理的衝突チェック(時刻 × 場所が他のイベントと重ならないか)
  ├ 論理的衝突チェック(ルーティンや事前の約束と矛盾しないか)
  └ 性格パラメータ注入AI からの情報を参考にした整合性判断
    (内省傾向が低いキャラが高い意志力を要する計画を連発している
     場合、採択率を控えめにする等)

Step 3: 採択された候補(0〜3 件)を構造化イベントとして生成
  ├ known_to_protagonist: true
  ├ source: "protagonist_plan"
  ├ 予想外度: 低
  └ 時刻・内容は Stage 1 の出力から引き継ぎ

Step 4: 翌日のイベント列に時刻順で挿入、衝突があれば時刻調整

Step 5: 主人公AI には調整過程を一切通知しない(隠蔽)
```

- **出力:** 更新された翌日のイベント列(週間イベントストアに書き戻す)

##### protagonist_plan イベントの扱い(脚本AI 生成イベントとの等価性)

採択されて週間イベントストアに挿入された `source: protagonist_plan` イベントは、**脚本AI が Day 0 Phase D で事前生成したイベントと完全に同じ扱い** で処理される。すなわち:

- 翌日の §4.2 本日のイベント列読み込みで、他のイベントと一緒に時刻順で読み込まれる
- 通常のイベントループ(§4.3 Perceiver → §4.4 動的活性化 → §4.6 RIM 並列処理)を通る
- 1 イベントパッケージとして §4.6d で行動履歴バッファに記録される

**実行/不履行の判断は統合エージェントが行う:**

§4.6 Step 3 Sub-step 3-4 の「source: protagonist_plan のイベントに対する追加指示」により、統合エージェントのプロンプトに以下が必ず含まれる:

```
「前日あなたは日記の中で『◯◯をする』と計画しました。
 実際にやるかやらないか、どのようにやるかは、今のあなたの
 気分・状況・優先順位で判断してください。」
```

つまり **「やると決めていた計画を今日の気分で実行する/しない/変更する」の判断が動的に行われる**。この判断は、理性ブランチの内面分析と衝動ブランチの反射反応の両方を踏まえて統合エージェントが最終決定する。

**不履行の場合の処理:**

統合エージェントが「やらない」または「できなかった」を選択した場合、その判断とその背後にある感情・理由が最終行動として決定される。次の §4.6 Step 4 情景描写・後日譚生成エージェントがそれを受けて、以下のような描写を付与する:

- 「やると決めていたはずだったが、実際には手を付けなかった」
- 「気が変わった」「気力が湧かなかった」「他のことに時間を使ってしまった」
- 「やろうとしたが途中で投げ出した」

など、不履行の質感が情景描写・後日譚の中に自然に織り込まれる。続いて §4.6c 価値観違反チェックエージェントが、理想自己・義務自己との不一致を検出した場合は事後感情(罪悪感・自己嫌悪等)を追加する。

**重要な原則:** 不履行のための特別な処理経路は存在しない。すべて既存の §4.6 イベントループの仕組みで処理される。protagonist_plan イベントは「実行される保証のある計画」ではなく、「前日に自分で立てた予定情報を含むイベント」として扱われ、実行/不履行は統合エージェントが動的に判断する。

##### Day 1 の扱い

Day 1 の週間イベントストアには `source: protagonist_plan` のイベントが一切存在しない。これは Day 0 の段階では翌日予定追加エージェントがまだ走っていないため、必然的にそうなる。

これを問題と捉える必要はない。Day 1 は「まだ自分の意志で翌日を動かす経験がない初日」として自然に立ち上がる。Day 1 の夜に翌日予定追加エージェントが初めて動き、Day 2 以降に protagonist_plan イベントが出現しうる構造で問題ない。

##### Day 7 の扱い

Day 7 の日記生成後も Stage 1 + Stage 2 を実行する。Day 8 は実際には来ないが、日記の最後に「明日はこうしたい」という言及が自然に入り、7 日間で物語が完全収束せず「続きがある」余韻が残る。Day 8 のイベント列は生成されるが実行されずに捨てられる。

Day 7 の日記生成後も Stage 1 + Stage 2 を実行する。Day 8 は実際には来ないが、日記の最後に「明日はこうしたい」という言及が自然に入り、7 日間で物語が完全収束せず「続きがある」余韻が残る。Day 8 のイベント列は生成されるが実行されずに捨てられる。

```

  └ Step 3: 翌日(Day N+1)の開始時の入力として保存
```

## 7. 未解決の論点

本節は、実装または提出前に検討・決定が必要な論点のみを列挙する。過去に論点として挙がったが既に解決済みのものは本節には含めず、**付録 A** の変更履歴で扱う。

### 7.1 運用フェーズ

- **「変容的体験」(L.A. Paul 的な大きな転機)を予想外度「高」で 1 回投入するか** ── 採用するなら Day 5 の山場と統合
- **価値観本体更新の累積閾値の具体値** ── 何日連続でどれくらいの不一致なら本体を動かすか

### 7.2 脚本AI 2 軸イベント生成の細部

- **分布制約の最適値** ── 予想外度「低」の下限を「半分以上」と仮置きしたが、最適な比率は実験で詰める必要あり
- **既知イベントと未知イベントの比率** ── 明示的な制約を入れるか(例:既知 3:未知 2)、LLM に任せるか
- **「なぜこのキャラに意味を持つか」の記述粒度** ── 1 文で十分か、3〜5 文の記述が必要か。粒度が粗すぎるとイベントとキャラの内面が接続されない
- **予想外度タグの LLM による解釈の安定性** ── 同じ「高」タグに対して、LLM が生成する感情質感が日によってブレる可能性。プロンプトで安定化を図る必要あり
- **7 日分一括生成のプロンプト容量** ── 7 日 × 4〜6 件 = 28〜42 件を 1 プロンプトで生成すると出力が肥大化する。分割生成と一括生成のトレードオフ
- **protagonist_plan 挿入による物語アークの乱れ** ── §4.9.4 で動的挿入されるイベントが、Phase D で設計された物語アーク(Day 5 山場への伏線)と衝突した場合の扱い

### 7.3 動的活性化方式の細部

- **活性化判定AI のコスト** ── 毎ステップで分類エージェントを呼ぶコストが許容範囲か
- **活性化パラメータ数の上限** ── 5〜10 個と仮置きしたが、最適な数は実験で詰める必要あり
- **活性化判定のキャッシング** ── 似たシーンで毎回判定し直すのは無駄。類似シーンの活性化結果を再利用する仕組みを入れるか

### 7.4 環境設計

- **「非日常の濃密な 7 日間」の物語骨格** ── 予想外度「高」の脚本AI に渡す世界設定をどの粒度で設計するか

### 7.5 実装コスト

- **統合LLM のコスト** ── Impulsive / Reflective の 2 レポート統合 + Higgins 照合 + 感情更新を 1 プロンプトで回すときのトークン数
- **動的活性化方式 + 統合LLM + 内省バリデーション の累積コスト** ── 全部入れると 1 日あたりの LLM コール数が膨らむ
- **理想実装版 vs 提出版のスコープ分け** ── 何を切るか、何を「説明資料で言及するだけ」にするか
- **全文ベタ貼り方式によるプロンプト肥大化** ── 自伝的エピソード + key memory + 過去日記本文を全部同梱することで、後半日(Day 6〜Day 7)のプロンプトが肥大化する。コスト見積もりが必要

### 7.6 Perceiver / 隠蔽まわり

- **気質に由来する行動を主人公AI が事後的にどう説明するか** ── 気質が「衝動的に上司に言い返した」という行動を生成したとき、主人公AI はそれを見て「なぜ言い返したんだろう」と考える。Nisbett & Wilson の示した「人は間違った理由を作る」現象を意図的に模す場合、主人公AI の事後説明と裏方エージェントが見ていた実際の気質との **ズレ** を評価指標に含めるべきか
- **性格パラメータの一部(#44 自己志向性・#48 内省傾向など)を例外的に主人公AI にも見せるか** ── メタ認知的に本人が自認しうるパラメータは隠蔽を緩和してもよい可能性がある。初版は一律隠蔽でスタートするが、実験で緩和する余地あり

### 7.7 LLM エージェンティック内省 + バリデーション方式の細部

- **バリデーションAI の検査粒度** ── 「#48 低キャラの日記に内省的な文が過剰」を検出する具体的な基準
- **内省含有率のキャラ整合性の定量化** ── 「整合している」「していない」を判断する閾値をどう設定するか
- **内省メモと日記の整合性チェック** ── 内省メモで言及した気づきが日記に反映されているかをチェックするか

### 7.8 記憶系の細部

- **key memory 抽出の字数制約** ── 300 字以内を守らせる制約と、字数オーバー時の再生成ロジックの詳細
- **段階圧縮のプロンプト設計** ── 各段階で何を残し何を捨てるかのプロンプト指示の最適化
- **key memory が 7 日目で 7 個になる想定の是非** ── 「本当に重要だった」が 7 日連続で必ず 1 個ずつ出るのは不自然では、という論点

## 9. このアーキテクチャの独自性(説明資料の差別化点)

提出資料で主張すべき、他のアプローチが持っていない設計上の独自性:

1. **【最重要】主人公AI と裏方エージェントの構造的分離による implicit / explicit 非対称の実装** ── 主人公AI(日記を書く主体)は自分の気質・性格層を直接参照できない。これを読めるのは Perceiver/Impulsive Agent(裏方)だけで、主人公AI には自然言語化された反射反応のみが届く。これは Nisbett & Wilson (1977) の「人間は自分の内的過程に直接アクセスできない」、McClelland et al. (1989) の「implicit motive と explicit motive は相関しない」という古典的知見の計算論的実装。他のキャラ LLM 研究では、キャラは自分のパラメータを全部知っているという前提で動いているが、本アーキテクチャはそれを明示的に否定する。この帰結として、(a) 日記生成プロンプトから「あなたは NS 高・HA 低で…」といった自己記述が原理的に消える、(b) 内省が「行動からの自己発見」プロセスになる(Bem Self-Perception)、(c) 主人公AI の自己説明と実際の気質との間に意図的な confabulation(事後でっち上げ)が生まれる、という 3 つの効果が同時に得られる

2. **気質・性格層と規範層の完全分離** ── 多くのキャラ LLM 研究は両者を混ぜているが、Parks-Leduc et al. (2015) のメタ分析的に両者は独立であるべき。「実際の being」と「判断基準の ought」を構造的に分けることで、内省の質が劇的に多様化する

3. **内部状態の 4 層構造(マクロ / 自伝的エピソード / 気質・性格 / 規範)** ── 動的活性化の対象外となる上位 2 層と、動的活性化の対象となる下位 2 層を構造的に分離。マクロ層は言語的指紋や秘密を常時同梱する。自伝的エピソード層も常時全文同梱方式(retrieval 不使用)。これに可視性の非対称が重なる(気質・性格層だけが主人公AI から隠蔽される)

4. **Phase A-3 自伝的エピソード生成と redemption bias 対策** ── 脚本AI が Phase A-1/A-2 から決定的エピソードを 5〜8 個生成し、McAdams の 5 カテゴリで構成を強制する。Nature 2026 の研究が示す LLM のデフォルト redemption bias を、プロンプトレベルで構造的に抑制する。価値観や夢の根に必ず「経験的な物語」が存在する状態を作る

5. **内省フェーズの 3 工程化(自己推測 + 過去統合 + 再解釈)** ── 内省は単なる「行動と価値観の照合」ではない。(a) 行動履歴から自分の特性を推測する(Bem Self-Perception)、(b) 自伝的エピソード・短期記憶DB(key memory + 通常領域)と今日の経験を接続する、(c) 薄れた過去記憶を今日の経験を通じて再解釈する、という 3 工程を必須化。これがないと日記は「今日の出来事を今日の基準で評価するだけ」のフラットなものになる。Conway の Self-Memory System における retrieval-dependent reconstruction の実装でもある

6. **Reflective-Impulsive Model の構造的実装(エージェント分離版)** ── Strack & Deutsch (2004) の二重過程モデルを、LLM への自然言語記述としてではなく **エージェント分離による並列処理アーキテクチャ** として実装する。Impulsive system は気質・性格層にアクセスできる裏方エージェント、Reflective system は規範層のみにアクセスできる主人公AI 本体。この分離により、両系統が同じプロンプト内で混ざることが構造的に防がれる。Loewenstein (1996) の visceral influences は「Reflective 系の重みを下げる」形で吸収する

7. **脚本AI による 7 日分イベント列の一括事前生成(Phase D)+ 既知/未知 × 予想外度の 2 軸メタデータ + 「なぜこのキャラに意味を持つか」メタデータ必須化** ── 脚本AI が Day 0 の最終フェーズで、7 日分 × 各日 4〜6 件(合計 28〜42 件)のイベント列を物語アークを俯瞰した上で **一括生成** する。日次ループ中の新規イベント生成は一切行わない。各イベントには (a) `known_to_protagonist: true/false` による既知/未知の厳密分離、(b) `source`(routine / prior_appointment / protagonist_plan)による既知の内訳明示、(c) 予想外度タグ(低/中/高)、(d) 「なぜこのキャラの気質・価値観・自伝的エピソードと結びつくか」の明示的記述、が付与される。分布制約(「低」下限・「高」上限・Day 5 山場要件)をプロンプトで強制。**これにより Day 5 山場への伏線を Day 1〜4 に仕込む脚本的設計が Day 0 段階で可能になる**。これは Generative Agents(Park et al. 2023)には無い、本アーキテクチャ固有の物語設計手法

8. **Higgins 自己不一致評価の Ideal / Ought 分離** ── 統合 LLM が 2 レポートを統合する過程で、Higgins (1987) の Ideal 不一致(落胆系感情)と Ought 不一致(不安系感情)を **明示的に分離** したプロンプトで生成する。質的差を保持することで、「なんか後ろめたい感じ」の一次元感情への潰れを防ぐ

9. **価値観への三段階アプローチ:理性ブランチの濃密な内面分析 + 統合エージェントの 2 ルート予測 + 独立の違反チェックエージェント** ── (a) 理性ブランチ(§4.6 Step 2)が、濃密な内面分析と価値観・過去経験との接続づけに加え、**示唆と予測を明示的に** 出力する。(b) 統合エージェント(§4.6 Step 3)が、衝動ルート / 理性ルートそれぞれについて「起こりうる良いこと・悪いこと」を内部的に組み立てて行動決定の材料とする(β方式)。(c) 行動決定後、**価値観違反チェックエージェント**(§4.6c)が独立して違反を検出し、罪悪感と簡易内省を付加する(事後検出)。「価値観を貫けば幸せ」という単純構造を壊し、葛藤の射程を広げる

10. **52 パラメータ × 動的活性化方式** ── 全パラメータを保持しつつ、シーンごとに必要なものだけ発火させる。人間の自己複雑性理論(Linville)を計算論的に実装したもの

11. **嗜好的自己構築(Phase B)+ 擬似経験的自己構築(Phase C、本実装では保留)の二重構築思想** ── Day 0 で 2 種類の自己を構築する設計思想。一つはエコーチェンバーによる嗜好的自己(何に惹かれるか / Phase B)、もう一つは擬似経験による経験的自己(どう判断するか / Phase C)。前者は人格の色付けを、後者は人格の骨組みを与える。**本実装では Phase C は時間的制約により保留するが、設計思想としては残す**

12. **深掘り型クロール(Curiosity-Driven Crawling)** ── Phase B で採用。キャラ自身の好奇心に従って情報を深掘りしていくことで、同じ初期シードから始めても何に「気になる」と反応するかでクロール経路が分岐する。強化学習の好奇心駆動型探索(Pathak et al. 2017)を人格構築に応用したもの

13. **葛藤誘発型イベント生成 + 7 日間の葛藤強度アーク** ── 各日の出来事を独立に生成するのではなく、週間スケールの物語アーク(導入 → 揺らぎ → 選択 → 跳ね返り → 山場 → 余韻 → 収束)と葛藤強度の波形に従って、キャラの内面構造を読み取った上でイベントが生成される。一括事前生成方式により、アーク設計が Phase D の段階で保証される

14. **内省フェーズが価値観更新の唯一の窓口** ── 出来事 → 価値観の直接経路を排除し、必ず内省という認知プロセスを経由させる。Festinger の認知的不協和理論の実装。内省フェーズでの更新対象は価値観本体のみ(累積閾値超え時のみ動く)

15. **価値観の不変層と可変層の分離** ── 7 日間で動くべきものと動かないものを構造的に分離。人格の安定性研究と整合

16. **3 時制の内面** ── 過去(記憶)・現在(知覚フィルター)・未来(理想自己)を全部持つキャラ LLM は少ない

17. **省略を明示的に設計する** ── 「何を書かないか」も人格表現として設計する。LLM の網羅的記述傾向をプロンプト指示で抑制し、選択的脱落を多段階で連鎖させる

18. **key memory 抽出(日記生成 AI が 1 日 1 個・300 字以内の「本当に重要だった瞬間」を抽出、段階圧縮の対象外として保護)** ── 日記を書き終えた直後の日記生成 AI(その日最も深く出来事を咀嚼した主体)が、「今日本当に重要だった瞬間」を 1 つだけ 300 字以内で抽出する。抽出された key memory は 7 日間フル保持され、段階圧縮や書き換えの対象外となる。それ以外の出来事は通常領域で段階圧縮(当日 400〜600 字 → 1 日前 200〜300 字 → 2 日前 80〜150 字 → 3 日前以上 20〜30 字)によって時間とともに薄れていく。これは Kahneman の Peak-End Rule + Brown & Kulik の Flashbulb Memory 研究に沿った、人間の記憶システムの計算論的実装

19. **裏方エージェント出力の検証フェーズ(§4.6b)** ── Perceiver / Impulsive Agent の出力を .md ファイルとしてパイプライン化し、検証 AI(Sonnet デフォルト、3 周失敗時 Opus エスカレーション)が隠蔽原則違反を 3 つのツール(str_replace / rewrite_sentence / request_full_regenerate)で修正する

20. **ムードと瞬間感情の構造的分離** ── 感情を「状態として持続するもの」と捉える多くのキャラ LLM 研究に対し、本アーキテクチャは **持続するのはムード(PAD 3 次元)だけ**、**具体的感情(喜び・悲しみ・怒り等)は瞬間的に生じてログに残り消える** という構造を明示的に採用。感情カテゴリは LLM の描写語彙として参考知識プロンプトで供給される。Barrett の Constructed Emotion Theory の計算論的実装に近い

21. **統合エージェント内の 2 ルート予測(β 方式)+ 理性ブランチ側の予測の明示** ── 統合エージェントが衝動ブランチ / 理性ブランチの推すルートそれぞれについて「起こりうる良いこと」と「起こりうる悪いこと」を独立した枠として内部的に組み立てる。理性側ルートの予測は理性ブランチ(§4.6 Step 2)も出力するので、統合エージェントはそれを参照材料として使う。両面性を LLM に明示的に出力させることで、「価値観通りに動けばハッピー」の単純構造を壊し、最終決定を未来予測ベースの比較として一発出しで行う

22. **Peak-End Rule(Kahneman)に基づく 1 日のムード集約** ── 1 日のムード推移を単純平均するのではなく、ピーク時ムードと終端時ムードの重み付き合成で最終ムードを算出する。重み係数は気質で変調(Persistence 高はピーク寄り、NS 高は終端寄り)

23. **既知/未知の厳密分離による Perceiver / 統合エージェントの反応差別化** ── 各イベントに `known_to_protagonist: true/false` を必須タグとして付与し、既知イベントには `source`(routine / prior_appointment / protagonist_plan)も明示する。この厳密分離により、Perceiver では「朝から気になっていた面談」のような前兆感情が、統合エージェントでは「昨日自分で決めたことを今どう感じているか」のような自己関与感情が、それぞれ質的に異なる形で生成される。曖昧な中間状態を一切許容しないことで、下流処理の質感が構造的に保証される

24. **翌日予定追加エージェント ── キャラの能動性経路の実装** ── 日記生成後、主人公AI 自身が「明日やりたいこと」を最大 3 個出力する Stage 1(強制しない、無ければ無いと明示的に書く)と、裏方の整合性調整エージェントが採択・挿入する Stage 2(性格パラメータ注入AI から #48 などの自然言語化情報を受けて判断)の 2 段構成で、キャラが自分の意志で翌日の何かを動かす経路を物語構造に組み込む。採択された候補は翌日のイベント列に `source: protagonist_plan` で挿入され、脚本AI 生成イベントと完全に等価に扱われる。翌日の統合エージェントには「前日あなたは日記の中で『◯◯をする』と計画しました。実際にやるかやらないか、どのようにやるかは、今のあなたの気分・状況・優先順位で判断してください」という追加指示が入るため、計画の実行 / 放棄 / 変更が動的に決まる。不履行の場合は情景描写・後日譚生成エージェントと価値観違反チェックエージェントが感情・罪悪感を自然に描写する。これは Generative Agents(Park et al. 2023)にも存在しない、本アーキテクチャ固有の差別化要素

25. **短期記憶DB の 2 領域構造(key memory 領域 + 通常領域の段階圧縮)** ── 記憶の重要度判定を数値スコアではなく「key memory 抽出」で、時間経過による薄れを「段階圧縮」で実装する。key memory は 7 日間フル保持、通常領域は 4 段階圧縮。これにより、retrieval 機構を使わない全文ベタ貼り方式でもプロンプトサイズが肥大化せず、かつ「本当に重要だった瞬間」は 7 日間保護されて主人公AI が何度も参照できる

26. **情景描写・後日譚生成エージェント** ── 行動決定エージェントの直後に配置される専用エージェント。「何をしたか」まで決まった後、濃密な情景描写(場の空気、周囲の反応、会話の具体、物理的環境、キャラ自身の外的な振る舞い)と後日譚(行動に対する他者の応答、直後の余波、次のイベントまでの間の背景的出来事)を付与する。既知イベントでは「予定通りだったか、予定がどう逸れたか」、未知イベントでは「突然性と立て直し」、protagonist_plan では「自分が決めたことを実際にやってみてどうだったか」の描写に重点が置かれる。これにより日記本文の物理的厚みと具体性が構造的に保証される

27. **理性ブランチの明示的予測機能** ── 理性ブランチの主務を「濃密な内面分析レポート + 示唆 + 予測」に変更し、これらをすべて **明示的にプロンプトで指示** する。従来の「価値観フィルター → トレードオフ分析」の受動的な構造ではなく、能動的に「この状況は自分にとって何を意味するか」「価値観・知識・過去経験との接続」「理性側ルートを取った場合の良いこと / 悪いことの予測」を書かせる。統合エージェントはこれを参照材料として、衝動側ルートとの比較を行う

- **理性ブランチの出力から「トレードオフ分析」を削除** ── 各選択肢の未来予測は統合エージェント側の 2 ルート予測に移管

### A.4 v9 → v10 主要変更(本版)

- **神AI(godAI)呼称の廃止と「脚本AI」への統一** ── Phase A-2 ミクロパラメータ生成、Phase A-3 自伝的エピソード生成、Phase D 7 日分イベント列一括事前生成、すべてを「脚本AI」が担当する形に用語統一
- **Phase D の全面復活と再設計** ── v8 で撤回されていた Phase D を、7 日分 × 各日 4〜6 件(合計 28〜42 件)のイベント列を脚本AI が物語アークを俯瞰した上で **一括事前生成** するフェーズとして全面再設計。日次ループ中の新規イベント生成は一切行わない方針に統一。Day 5 山場への伏線を Day 1〜4 に仕込む脚本的設計が Day 0 段階で可能になる
- **既知/未知の厳密分離** ── 各イベントに `known_to_protagonist: true/false` を必須タグとして付与。既知イベントには `source`(routine / prior_appointment / protagonist_plan)も明示。曖昧な中間状態を一切許容しない方針を確立
- **翌日予定追加エージェント(§4.9.4)の新設** ── 日記生成後、主人公AI 自身が「明日やりたいこと」を最大 3 個出力する Stage 1(強制なし)と、裏方の整合性調整エージェントが採択・挿入する Stage 2 の 2 段構成。採択された候補は翌日のイベント列に `source: protagonist_plan` で挿入される。キャラの能動性経路を物語構造に組み込んだ
- **Stage 1 を「最大 3 個、強制なし」方式に決定** ── 「必ず 3 個挙げる」の強制を採用せず、「思いつくものがあれば積極的に書く、無ければその状態を明示的に書く」方式に決定。キャラが「今は何も考えたくない」状態も自然に表現できる
- **Stage 2 に性格パラメータ注入AI の仕組みを導入** ── Stage 2 は裏方エージェントとして気質・性格層へのアクセス権を持つが、直接解釈せず、裏方の性格パラメータ注入AI が必要なパラメータ(特に #48 内省傾向)を自然言語化して Stage 2 に注入する形式を採用。これにより「内省傾向の低いキャラが高い意志力を伴う計画を連発する」ような状況に対して、Stage 2 が自然言語情報に基づいて総合判断できるようになる
- **protagonist_plan イベントを脚本AI 生成イベントと完全に等価に扱う方針を確立** ── 翌日予定追加エージェントが挿入するイベントは通常のイベントループを通り、統合エージェントが動的に実行/不履行を判断する。不履行の場合は情景描写・後日譚生成エージェントと価値観違反チェックエージェントが感情・罪悪感を自然に描写する
- **Day 1 の protagonist_plan 不在を設計上の自然な状態として肯定** ── Day 0 の段階では翌日予定追加エージェントがまだ走っていないため、Day 1 の週間イベントストアには protagonist_plan イベントが存在しない。これを「まだ自分の意志で翌日を動かす経験がない初日」として受け入れ、Day 0 段階での事前挿入は行わない
- **経験DB の廃止と短期記憶DB の 2 領域構造への統合** ── v9 までの「経験DB」という独立概念を廃止。記憶系を (a) 自伝的エピソードDB(Phase A-3 生成、7 日間不変、全文ベタ貼り)、(b) 短期記憶DB(key memory 領域 + 通常領域)、(c) 日記本文ストア の 3 層に再整理
- **key memory 抽出の新設** ── 日記生成 AI が 1 日 1 個・300 字以内で「本当に重要だった瞬間」を抽出する独立ステップを新設。7 日間フル保持、段階圧縮対象外、書き換え不可
- **記憶圧縮エージェントの命名と独立化** ── 短期記憶通常領域の段階圧縮を担当する独立エージェントとして「記憶圧縮エージェント」を命名(§4.9.3.2)。key memory 抽出エージェントとは別立てのエージェント
- **短期記憶通常領域の段階圧縮方式の確立** ── 当日 400〜600 字 → 1 日前 200〜300 字 → 2 日前 80〜150 字 → 3 日前以上 20〜30 字の 4 段階で日ごとに要約度合いが深まる。初期日(Day 1〜Day 3)では該当する段が存在しない場合はスキップする
- **日記生成後処理の順序を「ムード更新 → key memory 抽出 → 記憶圧縮 → 翌日予定追加 → DB 更新」に確定** ── §4.9.2 Peak-End Rule + 日記整合補正で算出された日次集約ムードを、key memory 抽出エージェントと記憶圧縮エージェントが参照する形に統一。これは Kahneman の Peak-End Rule の趣旨(1 日を振り返ったときに何が残るか)と整合する
- **価値観の具体例リスト・行動指針リストの完全撤回** ── v9 まで規範層の可変サブモジュールとして維持されていた「価値観ごとの具体例明示リスト」「行動方針の明示リスト」を完全に撤回。規範層に可変サブモジュールは一切持たない方針に統一。Schwartz 骨格 + 動的活性化で十分と判断。明示リストを持つとキャラが「ルールに従って動くロボット」的になる副作用があるため、持たない方針を採る
- **擬似経験DB の廃止(Phase C 保留に伴う)** ── v9 まで Phase C で構築される想定だった「擬似経験DB」という独立概念を、Phase C の本実装保留に伴って本実装では作らない。設計思想としては残すが、本仕様書内の DB 一覧・入力リスト・プロンプト同梱ルールからは除外
- **retrieval 機構の全廃、全文ベタ貼り方式** ── 自伝的エピソードDB を含め、すべての記憶系参照を retrieval から全文ベタ貼りに変更。7 日間スパンならコスト的に許容できる。精度問題・検索失敗・ベクトル類似度の不安定性を構造的に回避
- **主人公AI の内部定義明確化(§0.4)** ── v9 まで曖昧だった「主人公AI」という呼称を、日次ループ内の判断・出力系の総称として明確化。Reflective system / 行動決定 / 情景描写・後日譚 / 内省 / 日記生成 / key memory 抽出 / 翌日予定 Stage 1 を含む
- **情景描写・後日譚生成エージェント(§4.6 Step 4)の新設** ── 行動決定エージェントの後段に配置される専用エージェント。濃密な情景描写 + 行動後の周辺的出来事を付与する
- **理性ブランチの主務変更** ── 「選択肢列挙 + 推奨行動」を主務から外し、「濃密な内面分析レポート + 示唆 + 予測を明示的にプロンプトで指示」に変更。自然に出る範囲任せではなく、明示的に予測まで書かせる
- **2 ルート予測の配置を β 方式に確定** ── 統合エージェントが内部的に両ルート予測を組み立てるのを基本とし、理性側ルートの予測は理性ブランチ(§4.6 Step 2)も出力する。統合エージェントはそれを参照材料として使う
- **行動決定の一発出し方式への統一** ── 10 選択肢方式や「自由生成 5 個 → 絞り込み 3 個 → 1 つ選ぶ」のハイブリッド方式を不採用。統合エージェントは 1 回の生成で最終行動を決める
- **Phase C(擬似経験構築)の明示的な保留** ── 本実装では作らない。説明資料では「理想実装版では中核フェーズ、提出版は時間的制約により保留」と明記
