# 日記エージェントの構造と安全機構

## エージェンティックループ構造
日記生成エージェントは `call_llm_agentic` / `call_llm_agentic_gemini` によるツール駆動の自律ループで動作する。

- **最大イテレーション**: 6回
- **ツール強制**: `tool_choice={"type": "any"}` でテキストのみ返答を禁止
- **終了条件**: `submit_` プレフィックスのツールが成功で呼ばれたとき

## 提供ツール
1. **`check_diary_rules`**: ドラフトの言語的指紋チェック（DiarySelfCritic経由）
2. **`submit_final_diary`**: 最終日記の提出

## 安全機構（2026-04-12 強化）
- `submit_final_diary` は `check_diary_rules` がSUCCESSを返していないと提出不可
- チェック済みドラフトと提出テキストが異なる場合、自動で強制チェック実行
- `diary_critic` が None（voice_fingerprint不在）でも、オーケストレーター側でAI臭い語彙・文字数の最低限ルールベースチェックを実施

## DiarySelfCritic（2026-04-12 LLMベース簡素化）
- ルールベースチェック（AI_SMELL_WORDSハードコード、_check_avoided_words等）を全廃止
- 1回のLLM呼び出しで全チェック（言語的指紋・避ける語彙・AI臭さ・文量・ムードPAD整合性・キャラクター整合性）を実行
- `corrected_diary` を廃止し `{"passed": bool, "issues": list[str]}` のみ返却。修正は主エージェントに委譲
- コンストラクタに `MacroProfile` を追加し、キャラクター基本情報を整合性チェック用にLLMへ渡す
- **設計原則**: 検証AIは「入力→照合→判定→フィードバック」のシンプルな構造。ルールベース複雑化やcritic自身による修正は行わない

## コンテキスト入力
user_message として以下を渡す:
- マクロプロフィール、世界設定、Day N の出来事、内省メモ、現在のムード(VAD)、記憶コンテキスト（key memory + 短期記憶DB）

## 記憶コンテキストの内容
`_build_memory_context()` は以下を結合:
- `key_memory_store.load_all()` → 重要記憶（別ファイルストア）
- `memory_db.normal_area` → 短期記憶DB
