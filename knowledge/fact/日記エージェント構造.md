# 日記エージェントの構造と安全機構

## エージェンティックループ構造
日記生成エージェントは `call_llm_agentic` / `call_llm_agentic_gemini` によるツール駆動の自律ループで動作する。

- **最大イテレーション**: 10回（第三者レビュー追加に伴い6→10に拡張）
- **ツール強制**: `tool_choice={"type": "any"}` でテキストのみ返答を禁止
- **終了条件**: `submit_` プレフィックスのツールが成功で呼ばれたとき

## 提供ツール
1. **`check_diary_rules`**: ドラフトの言語的指紋チェック（DiarySelfCritic経由）
2. **`third_party_review`**: 第三者（読者）視点での品質チェック（ThirdPartyReviewer経由）
   - check_diary_rules通過後のみ呼び出し可能
   - 理解可能性・面白さ・内部整合性・自然さ・イベント整合の5観点で評価
   - 不合格時は check_passed もリセットされ、修正後に両方やり直しが必要
3. **`submit_final_diary`**: 最終日記の提出

## 安全機構（2026-04-12 強化 + 第三者レビュー追加）
- `submit_final_diary` は `check_diary_rules` と `third_party_review` の両方がSUCCESSを返していないと提出不可
- チェック済みドラフトと提出テキストが異なる場合、自動で強制チェック実行
- `diary_critic` が None（voice_fingerprint不在）でも、オーケストレーター側でAI臭い語彙・文字数の最低限ルールベースチェックを実施

## DiarySelfCritic（2026-04-12 LLMベース簡素化）
- ルールベースチェック（AI_SMELL_WORDSハードコード、_check_avoided_words等）を全廃止
- 1回のLLM呼び出しで全チェック（言語的指紋・避ける語彙・AI臭さ・文量・ムードPAD整合性・キャラクター整合性）を実行
- `corrected_diary` を廃止し `{"passed": bool, "issues": list[str]}` のみ返却。修正は主エージェントに委譲
- コンストラクタに `MacroProfile` を追加し、キャラクター基本情報を整合性チェック用にLLMへ渡す
- **設計原則**: 検証AIは「入力→照合→判定→フィードバック」のシンプルな構造。ルールベース複雑化やcritic自身による修正は行わない

## ThirdPartyReviewer（2026-04-12 新設）
- `backend/agents/daily_loop/third_party_reviewer.py` に定義
- 第三者（初見の読者）として日記を評価するLLMベースの検証AI
- チェック観点: 理解可能性、面白さ・人間味、内部整合性、自然さ、イベントとの整合
- DiarySelfCritic（言語ルール）とは異なり、読者体験の品質を検証
- 戻り値形式は DiarySelfCritic と同一: `{"passed": bool, "issues": list[str]}`

## コンテキスト入力
user_message として以下を渡す（各セクションにwrap_contextによる説明付き）:
- マクロプロフィール、世界設定、Day N の出来事、内省メモ、現在のムード(VAD)、記憶コンテキスト（key memory + 短期記憶DB）

## コンテキスト説明付与（2026-04-12 新設）
- `backend/agents/context_descriptions.py` の `wrap_context()` ヘルパーで全エージェントに渡すコンテキストに「何のデータか・なぜ渡すか・どう使うか」の3点説明を付与
- 対象: daily_loop（5メソッド）、phase_a1、phase_a2、phase_a3、phase_d の全オーケストレーター
- エージェントロール別に説明を分岐（例: impulsive/reflective/diary で異なる使い方を指示）

## 記憶コンテキストの内容
`_build_memory_context()` は以下を結合:
- `key_memory_store.load_all()` → 重要記憶（別ファイルストア）
- `memory_db.normal_area` → 短期記憶DB
