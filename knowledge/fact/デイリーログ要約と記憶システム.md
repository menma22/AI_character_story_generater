# デイリーログ要約と記憶システム（Stage 18）

## 概要
2026-04-12にデイリーログの要約プロセスと記憶システムを全面的に再設計した。

## 変更前の問題
1. 翌日予定AIが日記生成の後に動作 → 日記に翌日への意向を反映できなかった
2. short_term_memoryのnormal_areaのソースが日記テキスト → 行動ログの要約ではなかった
3. 記憶圧縮がthree_plus_days_agoで200字打ち切り → LLM要約ではなかった
4. デイリーログに対する要約プロセスが未実装 → 忘却メカニズムが機能していなかった

## 新しい実行順序
```
内省 → 翌日予定 → 日記生成 → ムード更新 → key memory → デイリーログ保存＆要約 → ムードcarry-over
```

## 新しいフォルダ構造
```
daily_logs/
  day_01/
    001_full.json      ← 1日の全行動ログ（イベントサマリー + 内省 + ムード）
    002_summary.json   ← Day 1終了時の要約（~半分）
    003_summary.json   ← Day 3終了時の再要約（さらに半分）
  Day_1.md             ← 既存のMDアーカイブ（人間用、共存）
```

## エージェントへの渡し方
- 「短期記憶」= 全日のdaily_logsフォルダの最新IDファイル + key memoryを全て個別に渡す
- 「過去の日記」= diaries/フォルダから独立DBとして読み込み、参照用として渡す
- 「明日の予定」= 翌日予定AIの出力を日記プロンプトに追加

## 主な新規コンポーネント
- `DailyLogEntry` モデル（backend/models/memory.py）
- `DailyLogStore` クラス（backend/agents/daily_loop/orchestrator.py）
- `_create_daily_log_and_summarize()` メソッド
- `_build_past_diary_context()` メソッド

## next_day_planning.pyの変更
- `stage1_protagonist_plan()`のパラメータ: diary → events
- 日記生成の前に実行されるため、日記ではなくイベント処理結果から翌日計画を生成
